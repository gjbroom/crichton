* Unreleased (v0.3 development)

** Added

- *Amp orchestrator security alignment*: comprehensive hardening of the Amp CLI coding agent integration (cricht-qg7)
  - Config gate: Amp is now disabled by default; enable with =[amp] enable = true= in config.toml
  - Repo allowlist: =allowed_repo_roots= restricts which directories Amp can operate in
  - Credential mediation: API key resolved from encrypted credential store (=amp-api=), passed via minimal subprocess environment (HOME, PATH, AMP_API_KEY only)
  - No shell execution: all subprocess invocations use direct argv (no =/bin/sh -c=); =amp_test= tool now takes =test_runner= + =test_args= instead of freeform shell command
  - Structured I/O: git status tracking shows changed files, ANSI escapes stripped, output truncated with configurable caps
  - Audit logging: structured JSON audit events for every invocation attempt (repo, sanitized argv, policy decisions, exit code, elapsed time, changed files)
  - Restarts: =use-different-repo=, =retry-with-longer-timeout=, =return-partial-result= available in the SLIME/SLY debugger
  - JSON tools: new =amp_check_json=, =amp_code_json=, =amp_test_json= tools for programmatic chaining
- *TUI multiline input*: input area now wraps long prompts onto multiple lines instead of scrolling horizontally; grows and shrinks dynamically as you type; up/down arrow keys navigate between visual lines
- *LLM rate-limit retry*: When the API returns HTTP 429, Crichton now automatically retries up to 3 times with 15/30/45-second backoff before reporting failure to the TUI
- *RSS feed retry*: RSS monitor polls now retry transient HTTP failures up to 2 times before logging a warning
- *Debugger restarts*: key operations (feed fetch, skill invocation, API key resolution, pipeline steps, tool dispatch) now offer named restarts (=:retry=, =:use-value=, =:skip-step=, etc.) visible from SLIME/SLY debugger
- *AC power detection*: battery reports and status plists now include AC adapter online status via =ac-adapter-online-p=
- *TUI client* (Phase 1 MVP): new =crichton-tui= binary for interactive terminal use with full ANSI colour, adaptive layout, scrollable chat history with viewport, markdown rendering of LLM responses, input editing with cursor support, spinner during thinking, and local commands (=:quit=, =:clear=, =:new=, =:session=, =:status=). Uses cl-tuition (TEA/Elm architecture), no ncurses dependency. One-shot CLI mode also supported. (cricht-5bg, cricht-gu8, cricht-0ix, cricht-7po, cricht-317, cricht-k63, cricht-o5s, cricht-knm, cricht-y2d, cricht-hew, cricht-qhb)
- *TUI push notifications* (Phase 2): daemon pushes real-time notifications to subscribed TUI clients (cricht-531, cricht-72b)
  - =subscribe= RPC op: clients register for push notifications on connect
  - Notification registry: thread-safe subscriber tracking with per-stream write locks
  - Battery monitor posts notifications when thresholds are crossed
  - RSS monitor posts notifications when new items are found
  - Toast overlay: notifications appear as styled banners between title bar and chat
  - Auto-dismiss after 10 seconds, manual dismiss with Esc key
  - =:notifications= command shows notification history in chat
  - Notification history capped at 100 entries
- *Org-mode skill*: new built-in skill for reading, searching, creating, and appending to org-mode files with org-roam integration (cricht-1gm)
  - Parses org structure: headlines (TODO/priority/tags), property drawers, file keywords, ID links
  - Queries org-roam SQLite database: search by title/tag, backlinks, forward links, tag listing
  - Create org-roam notes with auto-generated timestamp filenames, UUID :ID: properties, filetags
  - Append to existing files (at end or under a specific headline)
  - LOCAL-ONLY: gated by =[orgmode]= config section; requires explicit =allowed_paths= (empty = deny all)
  - All DB results filtered by allowed paths; DB opened read-only
  - =orgmode= agent tool with actions: read, search, list_tags, backlinks, create_note, append, list_files, status
- *RSS monitor keyword filtering*: RSS monitors now accept =keywords=, =match_mode=, and =search_fields= parameters to automatically filter new items through the rss-filter WASM skill (cricht-8u0)
  - Filter config persists to storage and survives daemon restarts
  - When keywords are set, only matching items trigger notifications
  - Available via the =rss= agent tool's =monitor_start= action
  - No manual pipeline setup required — filtering is built into the monitor
- Skill parameter passing: =invoke-skill= now accepts =:params= and =:abi= keywords (cricht-u1x)
  - JSON ABI auto-selected when params provided or manifest declares =abi = "json"=
  - Agent =skills= tool =invoke= action passes =params= field through to WASM skills
  - Backwards compatible: zero-arg skills continue to work unchanged
- Manifest function metadata: =manifest-function-info=, =manifest-function-input-type=, =manifest-function-output-type=, =manifest-function-description= accessors for =[provides.function.*]= sections
- Skill pipeline executor: chain multiple skill invocations with cross-step references
  - =execute-pipeline= runs ordered steps (WASM skills and/or built-in functions)
  - Reference resolution via ={"ref": "step_id.key.subkey"}= objects in params
  - Built-in pipeline functions: =rss_fetch=, =rss_check=, =weather=
  - =pipeline-error= condition carries step-id and skill context
- Skills agent tool =pipeline= action: compose and run multi-step pipelines from the LLM
- *TUI streaming responses* (Phase 3): LLM responses now stream to the TUI in real-time as the model generates them (cricht-p19)
  - Anthropic =stream-message= implementation: SSE event parsing with =stream: true= API flag
  - Streaming agent loop: =run-agent/stream= streams text deltas via callback on every LLM call
  - =chat_delta= / =chat_done= RPC push messages: daemon pushes incremental text to connected TUI clients
  - Streaming request opt-in: TUI sends ={stream: true}= in chat requests; non-streaming clients unaffected
  - Real-time text rendering: assistant responses appear character-by-character with live viewport updates
  - Status bar shows "Streaming..." during active response delivery
  - Duplicate response suppression: streaming IDs tracked to prevent double-display from legacy response path
  - Graceful fallback: if streaming unsupported, falls back to non-streaming =send-message= automatically
- *User state: journal & memory tools* (Phase 2): agent can now write and search its own memory
  - =memory_write= tool: write curated facts to MEMORY.org or session notes to today's journal
  - =memory_search= tool: search journal entries by keyword across recent days
  - Daily journal files in =~/.crichton/state/journal/YYYY-MM-DD.org= (append-only, timestamped)
  - Pre-compaction journal flush: session history can be dumped to journal before context pruning
- *Sixel graphics support* (Phase 3): infrastructure for terminal image rendering (cricht-pmg)
  - Sixel terminal detection heuristic (foot, xterm, mlterm, wezterm, yaft)
  - =img2sixel= integration with result caching
  - Markdown image reference extraction (=![alt](path)=)
  - =:sixel on/off= toggle command
  - =:img path= command to display images in chat (placeholder mode; full sixel rendering in Phase 4)
- *Raindrop.io bookmark management*: new built-in skill and =raindrop= agent tool for managing bookmarks via the Raindrop.io API
  - =save=: create bookmarks with title, tags, description, and target collection
  - =search=: full-text search across all bookmarks with pagination
  - =list=: browse bookmarks by collection with sorting and pagination
  - =get=: retrieve a single bookmark by ID
  - =update=: modify bookmark title, tags, description, URL, collection, or favorite status
  - =delete=: move bookmarks to Trash
  - =collections=: list all collections (root and nested)
  - =create_collection=: create new collections with optional parent
  - =tags=: list all tags with usage counts
  - Authentication via OAuth2 test token stored as =raindrop-api= in credential store

** Changed

** Fixed

- *RSS monitors lost on restart*: RSS feed monitors registered via =rss-monitor-start= did not survive daemon restarts. Monitor configs are now persisted to storage and automatically restored during daemon init.
- *Battery monitoring lost on restart*: Battery monitoring enabled/interval state was not persisted. Now saved to storage and restored during daemon init; the config-based startup remains as defense-in-depth.
- *TUI streaming with tool use*: Multi-tool queries (e.g. "show a system dashboard") failed with Anthropic 400 errors because streaming SSE parser placed the text content block after tool_use blocks. Anthropic requires text before tool_use in assistant messages. Fixed content block ordering to match SSE stream order.
- *TUI stuck on "Streaming..."*: When the daemon encountered errors during streaming chat (rate limits, API errors, etc.), the TUI was never notified and remained stuck in "Streaming..." state indefinitely. Now pushes =chat_done= with error flag on all error paths.
- *Anthropic error messages lost*: Streaming API errors returned an opaque stream object instead of the actual error message. Now properly extracts the HTTP response body from dexador's error condition.
- rss-filter JSON parsing failures: plist params (from Lisp code and pipeline builtins) were serialized as JSON arrays instead of objects, causing WASM skills to fail with "failed to parse input JSON". Added =normalize-for-json= to recursively convert plists to hash-tables before serialization.
- rss-filter WASM skill panic: missing =#[link(wasm_import_module = "env")]= on Rust extern block caused =log()= host function to resolve to an abort stub instead of a proper WASM import (cricht-as0)

  - Usage metering via the existing token-usage system
- Persistent key-value store for WASM skills (cricht-mue, cricht-ms8, cricht-clx)
  - Per-skill isolated storage with age-encrypted JSON files (=~/.crichton/kv/<skill-id>.age=)
  - Write-through cache for performance, quota enforcement (keys, value size, total size)
- Persistent storage for daemon state (cricht-056)
  - Age-encrypted namespace-based storage (=~/.crichton/data/<namespace>.age=)
  - Token usage meters now persist across daemon restarts
  - RSS seen-item tracking now persists across daemon restarts
  - Automatic save on meter updates, load on daemon startup
  - Thread-safe operations with dirty-flag tracking for efficient flushing
  - Full CRUD API exposed as WASM host functions (=kv_get=, =kv_set=, =kv_delete=, =kv_exists=, =kv_list=)
  - Automatic cache preload at daemon startup, flush at shutdown
- Scheduled tasks now persist across daemon restarts (cricht-rlx, cricht-s51, cricht-6d1)
  - User-defined tasks (=user:*= prefix) automatically saved to encrypted storage
  - Restored on daemon startup with graceful handling of missing actions
  - Storage location: =~/.crichton/tasks/user-tasks.age=
- External WASM skill registry and management system (cricht-6d7)
  - Automatic discovery of skills in =~/.crichton/skills/= at daemon startup
  - On-demand loading and compilation of WASM skills (=skill.wasm= or =skill.wat=)
  - New =skills= agent tool with =list=, =info=, =invoke=, and =refresh= actions
  - Automatic registration of discovered skills as schedulable actions (=skill:<name>= prefix)
  - Skills can now be scheduled to run periodically via the scheduler tool

* v0.1 "Secure Core" - 2026-02-08

Initial release implementing all core security requirements from the design spec.

** Added

*** Daemon & Infrastructure
- SBCL-based daemon with systemd integration
- Microkernel architecture: daemon starts fresh via systemd, no save-lisp-and-die
- Minimal =crichton-client= binary for chat interface
- SLIME/SLY connectivity on port 4005 for live development
- RPC server on =~/.crichton/daemon.sock= using NDJSON over Unix sockets
- Configuration via =~/.crichton/config.toml=
- Structured JSON logging to =~/.crichton/logs/= with 0600 permissions

*** Security Features
- WASM skill sandbox using wasmtime (skills run in separate process)
- Capability manifests for skills (TOML format)
- Ed25519 code signing for skill verification
- Age-encrypted credential storage (=~/.crichton/credentials.age=)
- Age-encrypted session storage with configurable retention policy
- Network egress control (skills have no direct network access)
- Allowlist-based HTTP for skills via host functions

*** LLM Integration
- Anthropic Claude API client
- Tool/function calling support
- Agent loop with tool use
- Configurable model selection (default: =claude-sonnet-4-20250514=)

*** Built-in Skills
- *weather*: Ephemeris calculations (sunrise/sunset using USNO algorithm)
- *system-info*: CPU, memory, disk usage, temperature monitoring
- *timing*: Current time and timezone information
- *scheduler*: Periodic task execution (e.g., RSS feed monitoring)
- *rss*: RSS feed reading and monitoring
- *token-usage*: LLM API usage metering and reporting

All built-in skills follow dual interface pattern:
- Programmatic plist-returning functions
- Human-friendly formatted report functions

*** Commands
- =crichton-client= (interactive chat)
- =crichton-client "message"= (single-shot query)
- =systemctl --user start/stop/status crichton= (daemon management)

** Security

All 13 security findings from OpenClaw audit addressed:

- [CRITICAL] Skills isolated in WASM sandbox, separate process
- [HIGH] Credentials encrypted at rest with age
- [HIGH] Network egress controlled via allowlist
- [HIGH] Ed25519 code signing enforced
- [MEDIUM] Sessions encrypted and inaccessible to skills
- [MEDIUM] No shell interpretation in command execution
- [MEDIUM] Logs in user-private directory (0600)
- [MEDIUM] No skill marketplace (manual installation only)
- [LOW] Environment variables not exposed to skills
- [LOW] Capability manifests required for all skills

* Changelog Format

This changelog follows [[https://keepachangelog.com/][Keep a Changelog]] principles:

- Newest versions at the top (after Unreleased section)
- Each version includes release date in ISO format (YYYY-MM-DD)
- Changes grouped by type: Added, Changed, Fixed, Deprecated, Removed, Security
- User-focused language (not developer jargon)
- Reference BEADS issues where relevant (e.g., "Fixed cricht-xxx")

* Contributing to This Changelog

When completing work on a feature or bug fix:

1. Add an entry to the *Unreleased* section under the appropriate category
2. Write from the user's perspective ("Added readline support" not "Implemented ILineEditor")
3. Be concise but clear about what changed and why it matters
4. Reference the BEAD ID if applicable

When cutting a new release:

1. Rename "Unreleased" to the version number and add the release date
2. Create a new empty "Unreleased" section at the top
3. Review all entries for clarity and user focus
4. Update the git tag to reference the changelog
