* Overview

Crichton supports two types of skills:

1. *Built-in skills* — Written in Common Lisp, compiled into the daemon
2. *WASM skills* — Sandboxed WebAssembly modules loaded from =~/.crichton/skills/=

This guide covers *WASM skill development*, which provides isolation, portability, and security.

* Why WASM Skills?

- *Sandboxed*: No direct filesystem, network, or environment variable access
- *Portable*: Write once in any WASM-compatible language (Rust, C, AssemblyScript, WAT)
- *Signed*: Ed25519 signatures ensure code integrity
- *Capability-based*: Explicit manifest declares what the skill can do
- *Isolated*: Runs in separate process, cannot crash or compromise the daemon

* Skill Bundle Structure

A WASM skill is a directory in =~/.crichton/skills/= containing:

#+begin_src ditaa
~/.crichton/skills/my-skill/
+-- skill.toml          : Manifest (required)
+-- skill.wasm          : Compiled WASM binary (required)
+-- skill.wat           : WAT source (optional, for development)
#+end_src

** Manifest Format (skill.toml)

#+begin_src toml
[skill]
name = "my-skill"
version = "1.0.0"
author = "you@example.com"
description = "What this skill does"
signature = ""  # Ed25519 signature (optional for development)

[capabilities]
# HTTP domains this skill can access (empty = none)
http_domains = []

# Files this skill can read (not yet implemented)
read_files = []

# Files this skill can write (not yet implemented)
write_files = []

# Environment variables this skill can access (not yet implemented)
env_vars = []

# Resource limits
max_memory_mb = 64
max_cpu_seconds = 30
#+end_src

* Host Functions

Skills have NO direct system access. All capabilities come through *host functions* provided by Crichton.

** Available Host Functions

*** =env.log(level: i32, ptr: i32, len: i32)=

Log a message from WASM linear memory.

- *level*: 0=debug, 1=info, 2=warn, 3=error
- *ptr*: Byte offset in WASM memory where message string starts
- *len*: Length of message string in bytes (UTF-8)

*** =env.kv_get(key_ptr: i32, key_len: i32) -> i32=

Retrieve a value from the skill's key-value store.

- Returns: pointer to value in WASM memory, or -1 if not found
- *Note*: Currently returns 0 (success) or -1 (not found); value retrieval not yet implemented

*** =env.kv_set(key_ptr: i32, key_len: i32, val_ptr: i32, val_len: i32) -> i32=

Store a key-value pair in the skill's private storage.

- Returns: 1 on success, negative error code on failure

*** =env.http_request(method_ptr: i32, method_len: i32, url_ptr: i32, url_len: i32, headers_ptr: i32, headers_len: i32, body_ptr: i32, body_len: i32) -> i32=

Make an HTTP request (must be to an allowlisted domain).

- Returns: HTTP status code, or negative error code
- *Note*: Currently returns 403 (forbidden); full implementation pending

*** =env.get_secret(name_ptr: i32, name_len: i32) -> i32=

Retrieve a secret from Crichton's encrypted credential store.

- Returns: pointer to secret value, or -1 if not found/not allowed
- *Note*: Currently returns -1; full implementation pending

* Writing a Skill in WAT

WAT (WebAssembly Text format) is the human-readable representation of WASM.

** Minimal Example

#+begin_src wat
(module
  ;; Import host function
  (import "env" "log" (func $log (param i32 i32 i32)))

  ;; Export memory (required for host functions to read strings)
  (memory (export "memory") 1)

  ;; Data section: embed a string at offset 0
  (data (i32.const 0) "Hello from WASM!")

  ;; Export main function
  (func (export "main") (result i32)
    ;; Call log(1, 0, 16) — log level INFO, string at offset 0, length 16
    (call $log
      (i32.const 1)   ;; INFO level
      (i32.const 0)   ;; string starts at offset 0
      (i32.const 16)) ;; string is 16 bytes long

    ;; Return 42
    (i32.const 42)))
#+end_src

** Compiling WAT to WASM

Use =wasmtime= or =wat2wasm= from the WebAssembly Binary Toolkit (WABT):

#+begin_src bash
# Install wabt (if not already installed)
# Debian/Ubuntu: apt install wabt
# macOS: brew install wabt

# Compile WAT to WASM
wat2wasm skill.wat -o skill.wasm
#+end_src

Or use =wasmtime=:

#+begin_src bash
# wasmtime can compile on-the-fly, or use wat2wasm tool
wasmtime compile skill.wat -o skill.wasm
#+end_src

* Writing Skills in Other Languages

** Rust (with wasm32-unknown-unknown target)

#+begin_src rust
// No standard library in WASM
#![no_std]
#![no_main]

// Import host function
extern "C" {
    fn log(level: i32, ptr: *const u8, len: i32);
}

#[no_mangle]
pub extern "C" fn main() -> i32 {
    let msg = b"Hello from Rust!";
    unsafe {
        log(1, msg.as_ptr(), msg.len() as i32);
    }
    42
}

// Panic handler required for no_std
#[panic_handler]
fn panic(_info: &core::panic::PanicInfo) -> ! {
    loop {}
}
#+end_src

Compile:

#+begin_src bash
rustc --target wasm32-unknown-unknown -C opt-level=3 skill.rs -o skill.wasm
#+end_src

** C (with clang --target=wasm32)

#+begin_src c
// Import host function
__attribute__((import_module("env"), import_name("log")))
void log(int level, const char* ptr, int len);

__attribute__((export_name("main")))
int main(void) {
    const char* msg = "Hello from C!";
    log(1, msg, 13);
    return 42;
}
#+end_src

Compile:

#+begin_src bash
clang --target=wasm32 -nostdlib -Wl,--no-entry \
  -Wl,--export=main -o skill.wasm skill.c
#+end_src

* Code Signing

For production use, skills must be signed with an Ed25519 key.

** Generating a Signing Key

#+begin_src lisp
;; From SLIME/SLY REPL connected to daemon
(crichton/skills:generate-signing-keypair
  :private-key-path #p"~/.crichton/signing-key.priv")
#+end_src

Returns =(values key-id private-key-path public-key-path)=.

The public key is automatically installed in =~/.crichton/trusted-keys/{key-id}.pub=.

** Signing a Skill

#+begin_src lisp
;; From SLIME/SLY REPL
(let* ((wasm-bytes (uiop:read-file-string #p"~/.crichton/skills/my-skill/skill.wasm"))
       (signature (crichton/skills:sign-skill-bytes
                   wasm-bytes
                   #p"~/.crichton/signing-key.priv")))
  ;; Update skill.toml with signature
  (format t "signature = ~S~%" signature))
#+end_src

** Verifying a Skill

#+begin_src lisp
;; From SLIME/SLY REPL
(let ((manifest (crichton/skills:parse-skill-manifest
                  #p"~/.crichton/skills/my-skill/skill.toml"))
      (wasm-bytes (uiop:read-file-string
                    #p"~/.crichton/skills/my-skill/skill.wasm")))
  (multiple-value-bind (valid-p key-id)
      (crichton/skills:verify-skill-bundle manifest wasm-bytes)
    (if valid-p
        (format t "✓ Valid signature from key ~A~%" key-id)
        (format t "✗ Invalid signature~%"))))
#+end_src

* Testing Skills

** Manual Invocation via Runner

The WASM runner can be tested independently:

#+begin_src bash
# Start the runner (in one terminal)
sbcl --noinform --non-interactive \
  --eval '(require :asdf)' \
  --eval '(push #p"/home/gjbroom/devel/crichton/" asdf:*central-registry*)' \
  --eval '(asdf:load-system :crichton)' \
  --eval '(crichton/runner:main (list "--socket" "/tmp/runner.sock"))'
#+end_src

Send test request (in another terminal):

#+begin_src bash
# Use socat or netcat to send NDJSON
echo '{"id":1,"op":"invoke","wat":"(module (func (export \"main\") (result i32) (i32.const 42)))","export":"main"}' | socat - UNIX-CONNECT:/tmp/runner.sock
#+end_src

** REPL Testing

#+begin_src lisp
;; From SLIME/SLY REPL
(crichton/wasm:run-wasm-with-host-fns
  "(module
     (import \"env\" \"log\" (func $log (param i32 i32 i32)))
     (memory (export \"memory\") 1)
     (data (i32.const 0) \"test message\")
     (func (export \"main\") (result i32)
       (call $log (i32.const 1) (i32.const 0) (i32.const 12))
       (i32.const 42)))"
  "main")
;; => 42
#+end_src

* Skill Loading (Future Work)

*Note*: Automatic skill loading from =~/.crichton/skills/= is not yet implemented.

Currently, skills must be invoked explicitly via the runner RPC interface. Future versions will include:

- Automatic discovery of skill bundles in =~/.crichton/skills/=
- Registration with the agent loop as available tools
- Tool schema generation from manifest metadata
- Skill marketplace and installation commands

* Security Best Practices

1. *Start unsigned during development* — Add signatures only when publishing
2. *Minimize capabilities* — Request only the HTTP domains you actually need
3. *Validate inputs* — Skills receive untrusted input from the LLM
4. *Handle errors gracefully* — Return error codes, don't trap
5. *Keep secrets out of logs* — Never log API keys or credentials
6. *Test offline first* — Use KV storage and log functions before adding network access
