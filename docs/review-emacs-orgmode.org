#+TITLE: Code Review: Emacs Integration and Org-mode Skill
#+DATE: 2026-03-21
#+AUTHOR: Claude

* Overview

Code review of =emacs/crichton.el= (Emacs client) and
=skills/builtins/orgmode.lisp= (built-in org-mode skill).  Issues are
filed as beads; this document provides context and rationale.

* emacs/crichton.el

** BUG: =crichton--send-request= reverses alist field order  [cricht-ahx]

Lines 102–110.  The function builds an alist, pushes extra fields,
then calls =nreverse=:

#+begin_src emacs-lisp
(let* ((id (crichton--next-id))
       (msg `((id . ,id) (op . ,op))))
  (while fields
    (push (cons (pop fields) (pop fields)) msg))
  (crichton--send (nreverse msg))
  id)
#+end_src

Starting from =((id . 1) (op . "chat"))=, after pushing =(text
. "hi")=, msg is =((text . "hi") (id . 1) (op . "chat"))=.  After
=nreverse= the order is =((op . "chat") (id . 1) (text . "hi"))=.  The
=id= field ends up after =op=, and additional fields are reversed
relative to the call site.

JSON objects are semantically unordered, so this doesn't break the
protocol, but it is confusing and makes logs hard to read.  The intent
was presumably =(id op text ...)=.  Fix: build the extra-field list
separately and append before sending, rather than =push=/=nreverse=.

** STYLE: =crichton-reconnect-delay= defined but never used  [cricht-thv]

Lines 59–61.  The customization variable is declared and documented
but the sentinel (lines 169–173) does not schedule reconnection — it
just clears the process variable.  The presence of the variable implies
auto-reconnect behaviour that doesn't exist, which is misleading.

Either implement auto-reconnect using this variable or remove it.

** BUG: =crichton--handle-emacs-invoke= doesn't guard against missing =function= key  [cricht-3a3]

Lines 361–383.  The handler does:

#+begin_src emacs-lisp
(let* ((id (alist-get 'id msg))
       (func-name (intern (alist-get 'function msg)))
       ...)
#+end_src

If the daemon sends an =emacs_invoke= message without a =function=
field, =(alist-get 'function msg)= returns =nil=, and =(intern nil)=
signals a =wrong-type-argument= error.  The error is not caught here,
so it propagates up through the process filter, which does catch errors
— but the =id= in the response will be the id from the original message,
and no error reply is sent back to the daemon.

Add a nil check before calling =intern= and send an explicit
=not_allowed= or =bad_request= error if =function= is absent.

** REVIEW: Double-routing of responses in =crichton--dispatch=  [no bead — design note]

Lines 144–151.  When a response arrives for a pending callback, the
code fires the callback *and* calls =crichton--handle-response=.
=crichton--handle-response= guards on =crichton--streaming-buffer=
being nil, so the duplicate doesn't cause visible problems today.
However, it adds conceptual overhead and could be a latent bug if
=crichton--streaming-buffer= is ever nil while a callback response
arrives.  Consider routing callbacks and the streaming fallback as
mutually exclusive.

** REVIEW: =cat= process dependency in =crichton--chat-buffer=

Lines 275–283.  The comint-mode setup starts a =cat= subprocess as a
dummy process.  This works on POSIX systems but creates an unnecessary
subprocess.  Emacs 29+ provides =make-pipe-process= which avoids the
external dependency.  Consider using it as an alternative when
available.

* skills/builtins/orgmode.lisp

** BUG: =org-roam-list-tags= does not filter by allowed paths  [cricht-xaj]

Lines 676–693.  The docstring and an inline comment both claim the
function filters tags to those from nodes under allowed paths.  The
implementation does not do this:

#+begin_src lisp
(defun org-roam-list-tags ()
  "Return all distinct tags from nodes under allowed paths."
  (with-org-roam-db (db)
    (let* ((sql "SELECT DISTINCT t.tag
                 FROM tags t
                 JOIN nodes n ON n.id = t.node_id
                 ORDER BY t.tag")
           ...)
      ;; No path filtering here or after
      ...)))
#+end_src

The SQL query fetches all tags from the entire database.  Other
query functions (=org-roam-search-nodes=, etc.) call
=%om-filter-by-allowed-paths= on results.  This one does not.

Tags from nodes in disallowed directories are exposed to callers.
If tag names themselves carry sensitive information (project names,
personal labels) this is a security/confidentiality issue.

Fix: either add a path filter in the SQL (=JOIN files f ON ... WHERE
f.file LIKE ?=) or run the results through a cross-reference with the
allowed-path set.

** BUG: =orgmode-set-todo= reads the file twice  [cricht-i0y]

Lines 1189–1192.  The function reads the file content with
=uiop:read-file-string= into =text=, then immediately calls
=parse-org-file= on the same path, which reads it from disk again:

#+begin_src lisp
(let* ((text (uiop:read-file-string canonical))
       (lines (cl-ppcre:split "\\n" text))
       (lines-vec (coerce lines 'vector))
       (doc (parse-org-file canonical))   ; second read
       ...)
#+end_src

This is wasteful and, more importantly, creates a TOCTOU window: if
the file is modified between the two reads, =text= and =doc= will be
inconsistent.  =parse-org-file= should accept an optional text string
(or the caller should pass the already-read text), so the file is only
read once.

** BUG: =*headline-re*= hardcodes TODO keywords, diverges from =*todo-keywords*=  [cricht-6dp]

Lines 162–169.  The regex:

#+begin_src lisp
(defparameter *headline-re*
  (cl-ppcre:create-scanner
   "^(\\*+)\\s+(?:(TODO|DONE|NEXT|WAITING|CANCELLED|SOMEDAY)\\s+)?...")
#+end_src

and the parameter:

#+begin_src lisp
(defparameter *todo-keywords*
  '("TODO" "DONE" "NEXT" "WAITING" "CANCELLED" "SOMEDAY"))
#+end_src

are kept in sync manually.  They will diverge if one is updated
without the other.  Adding a custom keyword to =*todo-keywords*=
(e.g. for a =#+TODO:= declaration) will not make it recognised by
the parser.

Fix: generate =*headline-re*= from =*todo-keywords*= at definition
time, or consolidate to a single source.

** BUG: =%om-todo-from-db= produces contradictory WHERE when state is a done keyword  [cricht-bhs]

Lines 1011–1053.  When =include-done= is NIL (the default) and =state=
is supplied as a done keyword (e.g. ="DONE"=), the generated WHERE
clause will be:

#+begin_example
n.todo NOT IN ('DONE','CANCELLED') AND n.todo = 'DONE'
#+end_example

This is a contradiction; the query returns zero rows.  The function
offers no warning or error.

When =state= is explicitly provided, the =NOT IN= exclusion clause
should be skipped (the explicit state filter is already sufficiently
restrictive).

** PERFORMANCE: =orgmode-backlinks= LIMIT not pushed to DB queries  [cricht-zmf]

Lines 875–893.  The =limit= keyword argument is accepted but not passed
down to =org-roam-backlinks= or =org-roam-forward-links=.  Both
functions issue uncapped SQL queries:

#+begin_src lisp
(defun org-roam-backlinks (node-id)
  ;; No LIMIT in the SQL:
  (let* ((sql "SELECT DISTINCT n.id, n.file, n.title, n.level
               FROM links l JOIN nodes n ..."))
    ...))
#+end_src

The limit is enforced only in the caller via =subseq= after all rows
have been fetched.  For nodes with many backlinks this is a needless
memory and query cost.  Pass =limit= through to the SQL.

** ROBUSTNESS: =parse-org-preamble= hard-codes 50-line scan limit  [cricht-8md]

Line 338:

#+begin_src lisp
while (and line (< line-num 50))
#+end_src

Files with large PROPERTIES drawers, multiple =#+SETUPFILE= lines, or
lengthy comment headers before the =#+TITLE:= will silently yield
incomplete preamble data (title, filetags, or ID may be missed).  The
50-line limit is an undocumented constraint.  Consider raising it (200
is safer) or scanning until the first headline is hit regardless of
line count — the headline stop condition already exists in the loop.
