* Overview

This document defines the complete set of capabilities available to Crichton skills.
Each capability represents a permission to access specific system resources or
perform specific operations.

*Design Principles:*
1. *Least Privilege* — Skills get only what they declare and need
2. *Explicit Declaration* — All capabilities must be in manifest
3. *Defense in Depth* — Multiple layers verify capability usage
4. *Auditability* — All capability use is logged
5. *Revocability* — Capabilities can be restricted per-skill via policy

*Capability Model:*
- Declared in =skill.toml= manifest
- Enforced at load time (static analysis)
- Enforced at runtime (host function checks)
- Logged for audit (all uses recorded)

* Capability Categories

** I. Core Capabilities (v0.1 - Implemented)

Fundamental capabilities present in initial release.

*** 1. Logging (=logging=)

*Purpose:* Write structured log messages visible to user and daemon logs.

*Host Functions:*
- =log(level: i32, message_ptr: i32, message_len: i32)=
  - Levels: 0=debug, 1=info, 2=warn, 3=error

*Manifest Declaration:*
#+begin_src toml
[capabilities]
logging = true  # Boolean flag
#+end_src

*Security Properties:*
- Write-only (skills cannot read logs)
- Rate-limited (prevent log flooding DOS)
- Redacted output (secrets stripped by log4cl)
- Isolated (skills can't see other skills' logs)

*Risks:*
- Low — Log spam (mitigated by rate limiting)
- Low — Information disclosure (mitigated by redaction)

*Status:* ✅ Implemented (=crichton/wasm:host-log-callback=)

*** 2. Key-Value Storage (=storage.kv=)

*Purpose:* Persistent skill-scoped storage for configuration and state.

*Host Functions:*
- =kv_get(key_ptr: i32, key_len: i32) -> i32= (returns value ptr or -1)
- =kv_set(key_ptr: i32, key_len: i32, val_ptr: i32, val_len: i32) -> i32=

*Manifest Declaration:*
#+begin_src toml
[capabilities.storage]
kv_max_keys = 100          # Maximum number of keys
kv_max_value_size_kb = 10  # Max size per value
kv_total_size_mb = 1       # Total storage quota
#+end_src

*Security Properties:*
- Skill-scoped namespace (skill A can't read skill B's data)
- Quota enforcement (prevent storage exhaustion)
- Encrypted at rest (age-encrypted, future)
- No cross-skill access (complete isolation)

*Risks:*
- Medium — Storage exhaustion (mitigated by quotas)
- Low — Sensitive data leakage (mitigated by encryption)

*Status:* ✅ Implemented (=crichton/wasm:host-kv-*-callback=)

*** 3. Network - HTTP/HTTPS (=network.http=)

*Purpose:* Make HTTP/HTTPS requests to external APIs.

*Host Functions:*
- =http_request(method_ptr, method_len, url_ptr, url_len, headers_ptr, headers_len, body_ptr, body_len) -> i32=

*Manifest Declaration:*
#+begin_src toml
[capabilities.network]
http_domains = [
  "api.example.com",
  "*.github.com"        # Wildcard subdomains
]
http_methods = ["GET", "POST"]  # Default: all methods
http_max_response_mb = 10        # Response size limit
http_timeout_seconds = 30        # Request timeout
#+end_src

*Security Properties:*
- Domain allowlist (only declared domains accessible)
- Method restrictions (e.g., read-only skills = GET only)
- TLS-only (HTTP rejected, only HTTPS allowed)
- Logged (every request recorded with skill name, URL, status)
- Rate-limited (prevent abuse)

*Risks:*
- High — Data exfiltration (mitigated by domain allowlist + logging)
- High — SSRF attacks (mitigated by allowlist, no internal IPs)
- Medium — Network abuse (mitigated by rate limiting)

*Status:* ⚠️ Partial (allowlist checked, logging incomplete)

*** 4. Secrets Access (=secrets=)

*Purpose:* Read credentials from age-encrypted credential store.

*Host Functions:*
- =get_secret(name_ptr: i32, name_len: i32) -> i32= (returns secret ptr or -1)

*Manifest Declaration:*
#+begin_src toml
[capabilities.secrets]
allowed_secrets = [
  "github-api-token",
  "openai-api-key"
]
#+end_src

*Security Properties:*
- Explicit declaration required (can't fish for secrets)
- Credential mediator (daemon resolves, skill never sees credential store)
- Logged (every access recorded)
- Revocable (secrets can be rotated without skill changes)

*Risks:*
- Critical — Secret exfiltration (mitigated by network allowlist)
- High — Unauthorized access (mitigated by declaration requirement)

*Status:* ⚠️ Partial (host function exists, not fully implemented)

** II. System Capabilities (v0.2-v0.3)

Capabilities for interacting with the host system.

*** 5. Filesystem - Read (=filesystem.read=)

*Purpose:* Read files from host filesystem.

*Host Functions:*
- =fs_read_file(path_ptr: i32, path_len: i32, out_ptr: i32) -> i32=
- =fs_list_directory(path_ptr: i32, path_len: i32) -> i32=
- =fs_stat(path_ptr: i32, path_len: i32) -> i32=

*Manifest Declaration:*
#+begin_src toml
[capabilities.filesystem]
read_paths = [
  "~/.config/myapp/*.conf",  # Glob patterns
  "/etc/hostname",            # Specific files
  "~/Documents/**/*.txt"      # Recursive globs
]
max_file_size_mb = 100
#+end_src

*Security Properties:*
- Path allowlist (only declared paths accessible)
- No path traversal (=../= rejected, paths canonicalized)
- Sandboxed (no access to =~/.crichton/=, =/etc/shadow=, etc.)
- Read-only (no write access without separate capability)
- Logged (every read recorded)

*Risks:*
- High — Sensitive file access (mitigated by allowlist)
- Medium — Information disclosure (mitigated by logging)
- Medium — Path traversal (mitigated by canonicalization)

*Status:* ❌ Not Implemented

*** 6. Filesystem - Write (=filesystem.write=)

*Purpose:* Write files to host filesystem.

*Host Functions:*
- =fs_write_file(path_ptr: i32, path_len: i32, data_ptr: i32, data_len: i32) -> i32=
- =fs_append_file(...)=
- =fs_create_directory(...)=
- =fs_delete_file(...)=

*Manifest Declaration:*
#+begin_src toml
[capabilities.filesystem]
write_paths = [
  "~/output/**/*",           # Scoped to specific directory
]
create_directories = true    # Allow mkdir
delete_files = false         # Deny deletion
max_write_size_mb = 100
#+end_src

*Security Properties:*
- Path allowlist (only declared paths writable)
- No path traversal
- Sandboxed (can't write to system directories)
- Logged (every write recorded)
- Quota enforcement (prevent disk exhaustion)

*Risks:*
- Critical — Data destruction (mitigated by path allowlist)
- High — Disk exhaustion (mitigated by quotas)
- High — Malicious file creation (mitigated by sandboxing)

*Status:* ❌ Not Implemented

*** 7. Time & Clock (=time=)

*Purpose:* Access system time and sleep/wait.

*Host Functions:*
- =time_now() -> i64= (Unix timestamp in milliseconds)
- =time_now_formatted(format_ptr, format_len) -> i32= (formatted string)
- =time_sleep_ms(duration_ms: i32)=

*Manifest Declaration:*
#+begin_src toml
[capabilities]
time = true
max_sleep_seconds = 60  # Prevent infinite sleeps
#+end_src

*Security Properties:*
- Read-only (can't set system time)
- Sleep limits (prevent blocking daemon)

*Risks:*
- Low — Timing attacks (minimal risk in this context)
- Low — Sleep DOS (mitigated by max_sleep limit)

*Status:* ❌ Not Implemented (skills currently have no time access)

*** 8. Random Number Generation (=random=)

*Purpose:* Cryptographically secure random numbers.

*Host Functions:*
- =random_bytes(out_ptr: i32, out_len: i32) -> i32=
- =random_u32() -> i32=

*Manifest Declaration:*
#+begin_src toml
[capabilities]
random = true
#+end_src

*Security Properties:*
- CSRNG source (=/dev/urandom= or equivalent)
- No seed access (can't weaken randomness)

*Risks:*
- Low — None (random number access is safe)

*Status:* ❌ Not Implemented

*** 13. Subprocess Execution (=exec=)

*Purpose:* Execute external commands.

*Host Functions:*
- =exec_command(cmd_ptr: i32, cmd_len: i32, args_ptr: i32) -> i32=

*Manifest Declaration:*
#+begin_src toml
[capabilities.exec]
allowed_commands = [
  "/usr/bin/ffmpeg",
  "/usr/local/bin/imagemagick"
]
allowed_args_patterns = [
  "^-i .+\\.mp4$",      # Regex patterns for arg validation
]

max_output_mb = 100
#+end_src

*Security Properties:*
- Command allowlist (specific binaries only)
- Argument validation (regex patterns)
- Timeout enforcement
- Output size limits
- No shell interpretation (direct exec, no =sh -c=)
- Sandboxed environment (restricted PATH, cleared env)

*Risks:*
- Critical — Arbitrary code execution (mitigated by allowlist)
- High — Command injection (mitigated by no shell, arg validation)
- High — Privilege escalation (mitigated by running as daemon user)

*Status:* ❌ Not Implemented (explicitly rejected for v0.1-v0.2)

*Rationale:* High risk. Most use cases better served by purpose-built host functions
(e.g., =image_convert= instead of =exec ffmpeg=).

** V. Explicitly Denied Capabilities

Capabilities that are intentionally NOT provided for security reasons.

*** Raw Network Sockets (=network.raw=)

*Why Denied:*
- High risk: SSRF, port scanning, protocol attacks
- Bypass of domain allowlist
- Complex to sandbox safely

*Alternative:* Use =network.http= with allowlist. For specific protocols,
request purpose-built host functions (e.g., =mqtt_publish=, =smtp_send=).

*** Arbitrary Code Execution (=exec.arbitrary=)

*Why Denied:*
- Critical risk: complete sandbox escape
- No way to safely validate arbitrary commands
- Defeats entire security model

*Alternative:* Request specific commands via =exec= with strict allowlist,
or implement as built-in skill (Lisp code, reviewed by maintainers).

*** Raw Memory Access (=memory.raw=)

*Why Denied:*
- WASM sandbox escape vector
- Can read daemon memory
- Breaks isolation guarantees

*Alternative:* Not needed. WASM linear memory is sufficient for all legitimate uses.

*** Daemon Control (=daemon.control=)

*Why Denied:*
- Skills should not control daemon lifecycle
- Risk of DOS (kill daemon)
- Risk of configuration tampering

*Alternative:* Daemon control is via =crichton-client= CLI only (out-of-band).

*** Arbitrary File System Access (=filesystem.unrestricted=)

*Why Denied:*
- Risk of accessing sensitive files (=~/.ssh/=, =~/.crichton/credentials.age=)
- Risk of modifying system files
- Risk of path traversal attacks

*Alternative:* Use =filesystem.read= and =filesystem.write= with explicit path allowlists.

*** Environment Variables (=environment=)

*Why Denied:*
- Information disclosure risk: env vars often contain sensitive paths, configs, and system topology
- Even with allowlist and redaction, hard to predict what's sensitive (e.g., =XDG_CONFIG_HOME= reveals user paths)
- Common pattern of secrets in env vars (=*_TOKEN=, =*_SECRET=) makes this dangerous
- Legitimate use cases covered by other capabilities

*Alternative:* Use =secrets= capability for credentials, =storage.kv= for configuration, =time= capability for locale/timezone.

*** System & Process Info (=system.info=)

*Why Denied:*
- Fingerprinting risk: hostname reveals identity/organization, CPU/memory/disk usage reveals system topology
- Information disclosure even when read-only
- Skills should not need this data; monitoring is daemon responsibility

*Alternative:* Keep as daemon-only built-in skill (not exposed to WASM). For monitoring needs, skills can use =notifications= capability to push alerts based on thresholds, rather than polling system state.

* Skill Authorization (Multi-User Scenarios)

*Orthogonal to Capabilities:* While capabilities define what a skill *can do*, authorization defines *who can use it*.

** Authorization Model

Skills can declare authorization requirements in their manifest. System policy can override these declarations.

** Manifest Declaration

#+begin_src toml
[skill]
name = "org-file-editor"
description = "Edit and manage Org-mode files"

# Authorization: who can invoke this skill
[skill.authorization]
mode = "owner-only"        # "owner-only", "all-users", or "allowed-users"
allowed_users = []         # For "allowed-users" mode: ["gjbroom", "alice"]
#+end_src

** Authorization Modes

- =owner-only= — Only the system owner (Crichton installer) can invoke
- =all-users= — Any authenticated user can invoke (low-risk skills)
- =allowed-users= — Explicit allowlist of usernames

** Policy Override

System administrator can override skill authorization globally:

#+begin_src toml
# ~/.crichton/policy.toml

[policy.skill_authorization]
# Override skill declarations (more restrictive only)
"org-file-editor" = ["gjbroom"]              # Owner only, even if skill says "all-users"
"system-monitor" = ["gjbroom", "brandon"]    # Brandon can monitor
"weather" = "*"                              # Anyone can use (wildcard)
#+end_src

** Implementation

Authorization checks happen *before* skill execution:

#+begin_src lisp
;; Pseudocode for skill invocation
(defun invoke-skill (skill-name args user-context)
  (let ((skill (gethash skill-name *loaded-skills*)))
    ;; 1. Check authorization FIRST
    (unless (authorized-user-p skill (user-context-id user-context))
      (log:warn "Unauthorized skill invocation"
                :skill skill-name
                :user (user-context-id user-context))
      (return-from invoke-skill
        (error "Permission denied")))  ; Generic error, no details

    ;; 2. Execute skill (capability checks happen during execution)
    (execute-skill skill args user-context)))
#+end_src

** Use Cases

| Skill | Authorization | Rationale |
|-------|---------------|-----------|
| =weather= | =all-users= | Read-only, public API, low risk |
| =system-info= | =owner-only= | Reveals system topology |
| =org-file-editor= | =owner-only= | Modifies personal files |
| =rss-reader= | =all-users= | Read-only, no sensitive data |
| =backup-manager= | =["gjbroom", "admin-bot"]= | Critical operations, limited trust |

** Multi-User Scenarios

When Discord connector or other multi-user interfaces are enabled:

1. *User identification*: Each connection has authenticated user context
2. *Skill invocation*: Authorization checked against user ID
3. *Audit logging*: All invocations logged with user attribution
4. *Generic errors*: Unauthorized users receive "Permission denied", no details

** Example: Discord Connector

#+begin_example
Brandon (Discord) → "Hey Crichton, edit my TODO.org file"
                 ↓
         invoke_skill("org-file-editor", ...)
                 ↓
         Check: user="brandon", skill auth=["gjbroom"]
                 ↓
         Result: "Permission denied" (logged: brandon attempted org-file-editor)


Gordon (Discord) → "Hey Crichton, edit my TODO.org file"
                ↓
        invoke_skill("org-file-editor", ...)
                ↓
        Check: user="gjbroom", skill auth=["gjbroom"]
                ↓
        Result: Success, file edited
#+end_example

** Design Principles

1. *Default deny*: Skills without authorization declaration default to =owner-only=
2. *Least privilege*: Grant minimum access required
3. *Policy override*: System policy always more restrictive (can't grant more access than manifest)
4. *Simplicity*: Per-skill ACLs only (not per-operation, avoids N×M complexity)
5. *Auditability*: All authorization failures logged with user attribution

** Future Enhancements (v0.5+)

- Group-based authorization (=groups = ["admin", "family"]=)
- Time-based restrictions (=allowed_hours = "09:00-17:00"=)
- Rate limiting per user (=max_invocations_per_hour = 10=)
- Delegation (owner temporarily grants access to another user)

** Status

*Current:* ❌ Not implemented (single-user mode, not urgent until Discord re-enabled)

*Planned:* v0.4-v0.5 (multi-user connector support)

* Capability Introspection Protection

*Critical Security Requirement:*

Skills and their capability manifests *MUST NOT* be enumerable by LLM clients or external users.

** Prohibited Operations

The following operations are explicitly forbidden in tool handlers and RPC endpoints:

- =list_skills_with_capabilities()= — Enumerating skills with their declared capabilities
- =get_skill_manifest(skill_name)= — Retrieving skill.toml manifest contents
- =query_skill_permissions(skill_name)= — Querying what a skill is allowed to do
- =describe_skill_capabilities()= — Describing capability details to LLM

** Rationale

Even without direct access to a capability, an LLM could infer sensitive system information by learning which skills exist and what they're allowed to do:

- "Skill 'backup' can write to =/mnt/nas/*=" → reveals NAS mount point and backup strategy
- "Skill 'work' reads from =~/Acme/*=" → reveals employer or client names
- "Skill 'email' has 'gmail-token' secret" → reveals email provider
- "Skill 'monitor' can read =/var/log/apache2/*=" → reveals web server software
- "Skill 'database' connects to postgres://proddb:5432" → reveals infrastructure

*Defense in depth:* An LLM should only know skill names, not their permissions or configurations.

** Allowed Operations

Skills can be invoked, but require exact name knowledge:

- =invoke_skill(name, args)= — ✅ Allowed (name must be known in advance)
- =list_skill_names()= — ⚠️ Allowed with caution (returns names only, no capabilities)

** Implementation Requirements

1. *Tool handlers* expose only skill invocation, not inspection
2. *Capability enforcement* happens silently:
   - On success: return result
   - On failure: return generic error ("permission denied"), not details
   - Never return: "skill X lacks filesystem.write capability"
3. *RPC endpoints* do not expose manifest contents to clients
4. *Logs* record capability denials, but errors to LLM are opaque

** Example: Safe vs. Unsafe

#+begin_src lisp
;; ❌ UNSAFE: Exposes capabilities to LLM
(defun list-skills-with-capabilities ()
  (loop for skill in *loaded-skills*
        collect (list (skill-name skill)
                      (skill-manifest skill))))  ; Leaks manifest

;; ✅ SAFE: Invocation only, no introspection
(defun invoke-skill (name args)
  (let ((skill (gethash name *loaded-skills*)))
    (if skill
        (execute-skill skill args)  ; Capability checks happen here
        (error "Unknown skill"))))  ; Generic error, no details
#+end_src

** Mitigation Summary

| Attack Vector | Mitigation |
|---------------|------------|
| LLM queries skill list | Return names only, no capabilities |
| LLM queries skill manifest | Operation not implemented |
| LLM infers caps from errors | Generic errors, detailed logs only |
| User inspects via CLI | Allowed (user has direct filesystem access anyway) |
| External API queries | RPC endpoints don't expose manifests |

This protection layer ensures that *knowledge of system configuration remains compartmentalized*, preventing an LLM from building a detailed system profile through capability enumeration.

* Capability Verification

** Static Verification (Load Time)

When a skill is loaded:

1. Parse WASM imports (=analyze-wasm-imports=)
2. Map imports to capabilities:
   - =env.log= → requires =logging=
   - =env.http_request= → requires =network.http=
   - =env.kv_get= → requires =storage.kv=
3. Verify manifest declares all required capabilities
4. Reject if mismatch (imports =http_request= but no =network.http= declared)

** Runtime Verification (Execution Time)

When a host function is called:

1. Check =skill-context= for capability
2. Verify resource limits (quotas, timeouts)
3. Log the operation
4. Execute if authorized, reject otherwise

** Post-Execution Audit (Review Time)

After skill execution:

1. Review logs for capability usage
2. Verify all uses were within declared limits
3. Flag suspicious patterns (e.g., excessive network requests)

* Capability Policy Enforcement

System administrators can restrict capabilities globally:

#+begin_src toml
# ~/.crichton/policy.toml

[policy.capabilities]
# Globally disabled capabilities
deny = ["exec", "hardware", "database"]

# Require approval for high-risk capabilities
require_approval = ["filesystem.write", "git"]

# Global resource limits (override skill manifests)
[policy.limits]
max_memory_mb = 256
max_cpu_seconds = 60
max_network_requests_per_minute = 60

# Blocked domains (even if skill declares them)
[policy.network]
blocked_domains = ["pastebin.com", "raw.githubusercontent.com"]
#+end_src

* Future: Component Model Capabilities

With WASM Component Model (v0.5+), capabilities become *WIT interfaces*:

#+begin_src wit
// Instead of boolean flags in TOML,
// skills import specific WIT interfaces

world skill {
  import wasi:logging/logging;
  import wasi:http/outgoing-handler;
  import crichton:storage/kv;
  import crichton:secrets/accessor;

  export run: func(args: list<string>) -> result<string, string>;
}
#+end_src

*Advantages:*
- Capabilities are first-class types (not just TOML flags)
- Interface versioning (=wasi:http/outgoing-handler@1.0.0=)
- Static verification is automatic (WASM Component Model tools)
- Fine-grained: can import =storage/kv= without =storage/sql=
