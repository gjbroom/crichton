* Introduction

Crichton is a personal AI agent daemon that runs quietly in the background on
your Linux machine.  You chat with it through a terminal, a TUI client, or
Emacs; it answers questions, monitors your environment, manages bookmarks, and
can delegate coding tasks to the Amp CLI agent.

What makes Crichton different from a simple chatbot wrapper:

- *Skills* — built-in and third-party capabilities wired directly into the LLM
  conversation (weather, RSS feeds, battery monitoring, home automation, etc.)
- *Encrypted credentials* — API keys stored on disk using =age= encryption, never
  in plain text
- *WASM sandbox* — third-party skill code runs in a WebAssembly sandbox with no
  network or filesystem access unless explicitly granted
- *Persistent memory* — sessions, scheduled tasks, and feed monitors survive
  daemon restarts
- *Streaming responses* — the TUI and Emacs client show the LLM response as it
  arrives, word by word

This guide is for someone comfortable at the Linux terminal who wants to install
and use Crichton.  You do not need to know Common Lisp to use it.

* Prerequisites

Before installing Crichton, you need the following on your system.

** Required

- *SBCL* (Steel Bank Common Lisp) — the Lisp runtime Crichton runs on.

  #+begin_src bash
  # Debian/Ubuntu
  sudo apt install sbcl

  # Fedora/RHEL
  sudo dnf install sbcl

  # Arch
  sudo pacman -S sbcl
  #+end_src

- *age* — the encryption tool used to protect credentials and sessions.

  #+begin_src bash
  # Debian/Ubuntu (Debian 12+, Ubuntu 22.04+)
  sudo apt install age

  # Or install from source / GitHub releases:
  # https://github.com/FiloSottile/age/releases
  #+end_src

- *systemd* (user session) — Crichton runs as a systemd user service.  This is
  standard on most modern Linux desktops and servers.

- *An Anthropic API key* — Crichton uses Claude as its LLM.  Sign up at
  https://console.anthropic.com/ and create an API key.

** Optional (for specific skills)

- A *Pushover* account and app token (for push notifications)
- A *GitHub* personal access token (for GitHub integration)
- A *Raindrop.io* OAuth2 test token (for bookmark management)
- A *HOOBS* hub on your local network (for home automation)
- *Amp CLI* installed and on =PATH= (for coding orchestration)

* Installation

** 1. Get the code

#+begin_src bash
git clone https://github.com/gjbroom/crichton.git ~/devel/crichton
#+end_src

** 2. Build the client binary

The daemon itself is loaded fresh by systemd on each start — no binary needed
for the daemon.  But the command-line and TUI clients are compiled to standalone
executables so they start instantly.

Build the CLI client:

#+begin_src bash
sbcl --noinform --non-interactive \
  --eval '(require :asdf)' \
  --eval '(push #p"/home/YOUR_USERNAME/devel/crichton/" asdf:*central-registry*)' \
  --eval '(asdf:load-system :crichton-client)' \
  --eval "(sb-ext:save-lisp-and-die \"/home/YOUR_USERNAME/devel/crichton/crichton-client\" \
            :toplevel #'crichton-client:main :executable t :compression t)"
#+end_src

Replace =YOUR_USERNAME= with your actual username.

Build the TUI client:

#+begin_src bash
sbcl --noinform --non-interactive \
  --eval '(require :asdf)' \
  --eval '(push #p"/home/YOUR_USERNAME/devel/crichton/" asdf:*central-registry*)' \
  --eval '(asdf:load-system :crichton-tui)' \
  --eval "(sb-ext:save-lisp-and-die \"/home/YOUR_USERNAME/devel/crichton/crichton-tui\" \
            :toplevel #'crichton-tui:main :executable t :compression t)"
#+end_src

Each build takes a few minutes the first time while SBCL compiles dependencies.

** 3. Symlink the binaries

Put the binaries on your PATH:

#+begin_src bash
mkdir -p ~/bin
ln -sf ~/devel/crichton/crichton-client ~/bin/crichton-client
ln -sf ~/devel/crichton/crichton-tui    ~/bin/crichton-tui
#+end_src

Make sure =~/bin= is on your =PATH=.  Add this to your =~/.bashrc= or
=~/.zshrc= if needed:

#+begin_src bash
export PATH="$HOME/bin:$PATH"
#+end_src

** 4. Create the config directory

#+begin_src bash
mkdir -p ~/.crichton/logs
mkdir -p ~/.crichton/credentials
mkdir -p ~/.crichton/sessions
mkdir -p ~/.crichton/skills
#+end_src

** 5. Write a minimal config.toml

Create =~/.crichton/config.toml= with your basic settings:

#+begin_src toml
[llm]
provider = "anthropic"
model = "claude-sonnet-4-20250514"
api_key_credential = "anthropic-api-key"

[weather]
city = "Victoria"

[network]
egress_policy = "deny-all"
#+end_src

The =api_key_credential= value is the name of the credential entry you will
store in the next step.  You can leave it as =anthropic-api-key=.

** 6. Store your Anthropic API key

Crichton stores secrets using =age= encryption — your API key is never saved
in plain text.  To store it, you use the Lisp REPL directly:

#+begin_src bash
sbcl --noinform --non-interactive \
  --eval '(require :asdf)' \
  --eval '(push #p"/home/YOUR_USERNAME/devel/crichton/" asdf:*central-registry*)' \
  --eval '(asdf:load-system :crichton)' \
  --eval "(crichton/credentials:store-credential
           \"anthropic-api-key\"
           (list :key \"sk-ant-YOUR-KEY-HERE\"))" \
  --eval '(sb-ext:exit)'
#+end_src

Replace =sk-ant-YOUR-KEY-HERE= with your actual Anthropic API key.

To verify the credential was stored:

#+begin_src bash
ls ~/.crichton/credentials/
# Should show: anthropic-api-key.age
#+end_src

** 7. Install the systemd service

Edit =assets/crichton.service= to use your actual home directory and crichton
source path:

#+begin_src bash
cp ~/devel/crichton/assets/crichton.service ~/.config/systemd/user/
#+end_src

Open =~/.config/systemd/user/crichton.service= in your editor and update:

- =Environment=HOME==  — change to your actual home directory
- The path in =--eval "(push #p\"/home/gjbroom/devel/crichton/\" ..."= —
  change to your actual crichton source path

The file should look like this after editing (with your paths):

#+begin_src ini
[Unit]
Description=Crichton AI Agent Daemon
After=network.target

[Service]
Type=simple
Environment=HOME=/home/YOUR_USERNAME
ExecStart=/usr/bin/sbcl --noinform --non-interactive \
  --eval "(require :asdf)" \
  --eval "(push #p\"/home/YOUR_USERNAME/devel/crichton/\" asdf:*central-registry*)" \
  --eval "(asdf:load-system :crichton)" \
  --eval "(crichton/daemon:start-daemon :foreground t)"
ExecStop=/bin/kill -SIGTERM $MAINPID
Restart=on-failure
RestartSec=5

[Install]
WantedBy=default.target
#+end_src

Enable and start the daemon:

#+begin_src bash
systemctl --user daemon-reload
systemctl --user enable crichton
systemctl --user start crichton
#+end_src

The first start takes a minute or two while SBCL compiles the system.
Subsequent starts are faster.

** 8. Verify the installation

#+begin_src bash
# Check daemon status
systemctl --user status crichton

# Send a test message
crichton-client "Hello, are you there?"
#+end_src

You should see a response from the AI in a few seconds.

If something goes wrong, check the logs:

#+begin_src bash
journalctl --user -u crichton -f
#+end_src

All configuration lives in =~/.crichton/config.toml=.  The daemon reads it at
startup.  Most settings can be changed without restarting by asking the agent:
"Please reload your configuration."

Most configuration changes take effect without a daemon restart.  Ask the

* Storing Credentials

Credentials are stored in age-encrypted files under =~/.crichton/credentials/=.
Each credential is a named bundle of key-value fields (e.g., a token, username,
and password together).

** How to store a credential

Use SBCL directly from the command line.  This is a one-time setup step per
integration.

The general pattern is:

#+begin_src bash
sbcl --noinform --non-interactive \
  --eval '(require :asdf)' \
  --eval '(push #p"/home/YOUR_USERNAME/devel/crichton/" asdf:*central-registry*)' \
  --eval '(asdf:load-system :crichton)' \
  --eval '(STORE-CREDENTIAL-CALL-HERE)' \
  --eval '(sb-ext:exit)'
#+end_src

Specific credential commands for each integration are listed in the Built-in
Skills section below.

** Listing stored credentials

#+begin_src bash
sbcl --noinform --non-interactive \
  --eval '(require :asdf)' \
  --eval '(push #p"/home/YOUR_USERNAME/devel/crichton/" asdf:*central-registry*)' \
  --eval '(asdf:load-system :crichton)' \
  --eval '(print (crichton/credentials:list-credentials))' \
  --eval '(sb-ext:exit)'
#+end_src

** Updating a credential

Add =:overwrite t= to replace an existing credential:

#+begin_src lisp
(crichton/credentials:store-credential
 "credential-name"
 (list :field "new-value")
 :overwrite t)
#+end_src

* Using the Clients

** crichton-client (CLI)

The CLI client is the simplest way to talk to Crichton.

*** Single-shot mode

Pass your message as a command-line argument:

#+begin_src bash
crichton-client "What's the weather like?"
crichton-client "How much battery do I have left?"
crichton-client "Summarize what's new in my RSS feeds"
#+end_src

*** Interactive mode

Run without arguments for a readline-powered prompt:

#+begin_src bash
crichton-client
#+end_src

You will see a =>=  prompt.  Type your message and press Enter.  The session
continues until you type =:quit= (or =:q=, or =:exit=), or press Ctrl-D.

Command history is saved across sessions in =~/.crichton/history= (up to 1000
entries).  Use the Up/Down arrows to navigate history, and the standard Emacs
line-editing keys (Ctrl-A for start-of-line, Ctrl-E for end-of-line,
Ctrl-K to delete to end, etc.).

*** Daemon management

#+begin_src bash
systemctl --user start crichton    # start the daemon
systemctl --user stop crichton     # stop the daemon
systemctl --user restart crichton  # restart (needed after config changes to swank_port)
systemctl --user status crichton   # check if it's running
journalctl --user -u crichton -f   # follow the live logs
#+end_src

** crichton-tui (Terminal UI)

The TUI client is a full-screen terminal application with a scrollable chat
history, real-time streaming responses, and push notifications.

#+begin_src bash
crichton-tui
#+end_src

| Ctrl-C        | Press twice to quit (first press arms confirmation) |

Type these at the prompt and press Enter:

|------------------+-----------------------------------------------------|

*** Push notifications

The TUI automatically subscribes to push notifications from the daemon.
Battery alerts and RSS feed updates appear as banner overlays between the
title bar and chat area.  They dismiss automatically after 10 seconds, or
immediately when you press Escape.

*** Setup

Add to your Emacs init file:

(add-to-list 'load-path "/home/YOUR_USERNAME/devel/crichton/emacs")

* Built-in Skills

Built-in skills are capabilities baked into the daemon itself.  They run
directly in the daemon process (not in the WASM sandbox) and have access to
the network via the daemon's egress policy.

You interact with all of them by chatting naturally — the agent decides which
skills to call based on your message.

** Weather

Fetches current conditions and multi-day forecasts for Canadian cities from
the Environment Canada MSC GeoMet API.

*Configuration* (=~/.crichton/config.toml=):

#+begin_src toml
[weather]
city = "Victoria"   # Default city; override per-query by naming a city
#+end_src

*Network* — requires =api.weather.gc.ca= in =allowed_builtin_http_domains= if
you are using =egress_policy = "deny-all"=:

#+begin_src toml
[network]
egress_policy = "deny-all"
allowed_builtin_http_domains = ["api.weather.gc.ca"]
#+end_src

*Credentials* — none required.

*Example interactions*:

#+begin_example
What's the weather like today?
What's the forecast for Ottawa this week?
Any weather warnings in my area?
#+end_example

** System Information

Reports CPU load average, memory usage, disk usage, and thermal sensor
readings directly from =/proc= and =/sys=.  No external commands are used.

Can also run as a background monitor: Crichton checks the system periodically
and sends push notifications when any metric crosses a configured threshold.
Monitoring state (enabled/disabled and interval) is persisted so it survives
daemon restarts.

Works out of the box — no configuration or credentials required.

*Configuration* (=~/.crichton/config.toml=):

#+begin_src toml
[system]
mem_alert_percent = 90     # Alert when memory usage >= 90% (default)
cpu_alert_load = 2.0       # Alert when 1-minute load/CPU >= 2.0 (default)
disk_alert_percent = 90    # Alert when any disk >= 90% full (default)
temp_alert_celsius = 85    # Alert when any thermal zone >= 85°C (default)
interval = 300             # Check every 5 minutes (default)
#+end_src

Each alert fires once when the condition is first detected.  It will not
repeat until the metric recovers and then crosses the threshold again.

*Example interactions*:

#+begin_example
How loaded is my system right now?
How much free memory do I have?
Is my CPU running hot?
Start monitoring my system and alert me if anything looks bad.
Stop system monitoring.
#+end_example

** Timing and Time Zones

Reports the current time with microsecond-resolution and supports timing
measurements.  Useful for understanding when the daemon is responding and
for benchmarking.

Works out of the box — no configuration or credentials required.

*Example interactions*:

#+begin_example
What time is it?
What time is it in Tokyo right now?
How long did that last tool call take?
#+end_example

** Scheduler and Reminders

The scheduler runs tasks at specified intervals or at fixed times of day.
It backs the RSS monitor and battery monitor, and can also run WASM skills
on a schedule.

Scheduled tasks are persisted to encrypted storage and survive daemon
restarts.  User-defined tasks (those with names starting =user:=) are
automatically saved and restored on the next startup.  System tasks (battery
monitor, RSS monitors) are re-registered at startup from config — they do
not need persistence.

Works out of the box — no configuration required.

*Example interactions*:

#+begin_example
Check my RSS feeds every 30 minutes.
Remind me about my stand-up meeting every day at 9 AM.
What scheduled tasks are currently running?
Stop the RSS monitor.
#+end_example

*Persistence inspection*:

#+begin_example
Manually save my scheduled tasks to disk now.
Which of my scheduled tasks would be lost if the daemon restarted?
#+end_example

** Task Store

Provides encrypted persistence for user-defined scheduled tasks.  Tasks are
saved to =~/.crichton/tasks/user-tasks.age= and restored on daemon restart.

This operates transparently behind the Scheduler skill.  Under the hood:

- *What persists*: tasks whose names begin with =user:= (set automatically
  when you schedule via the agent tool, or manually with the =user:= prefix).
- *What doesn't*: built-in system tasks like =battery-monitor= and RSS monitors
  — these are re-registered from config on every start.
- *Storage*: =~/.crichton/tasks/user-tasks.age= (age-encrypted, 0600 permissions).
- *On restore*: past one-shot tasks are silently skipped.  Tasks whose backing
  action is no longer registered are skipped with a warning.

*Troubleshooting persistence*:

If you suspect tasks are not surviving restarts, check from the REPL or
=crichton-client --eval=:

#+begin_src bash
# See which persisted tasks would be skipped on next restart
crichton-client --eval '(crichton/skills:list-unrestorable-tasks)'

# Manually trigger a save without waiting for daemon shutdown
crichton-client --eval '(crichton/skills:persist-user-tasks)'

# Inspect the current live user tasks as readable JSON

* WASM Skills

WASM (WebAssembly) skills extend Crichton with third-party capabilities.  They
run in a strict sandbox: a separate process using the wasmtime runtime, with
no network access, no filesystem access, and no environment variables — unless
the skill's manifest explicitly declares those capabilities and Crichton
approves them.

Think of a WASM skill as a plugin in a locked room.  It receives input from
Crichton, computes a result, and returns it.  It cannot reach out to the
network or read your files unless that is declared upfront.

** Installing a WASM Skill

1. Obtain the skill directory.  It must contain at minimum:
   - =skill.toml= — the capability manifest
   - =skill.wasm= — the compiled WebAssembly binary

2. Copy the directory to =~/.crichton/skills/=:

   #+begin_src bash
   cp -r /path/to/skill-name ~/.crichton/skills/
   # Result: ~/.crichton/skills/skill-name/skill.toml
   #          ~/.crichton/skills/skill-name/skill.wasm
   #+end_src

3. Restart the daemon to pick it up:

   #+begin_src bash
   systemctl --user restart crichton
   #+end_src

4. Verify the skill was loaded:

   #+begin_example
   List the available skills.
   #+end_example

** The skill manifest (skill.toml)

Every WASM skill has a =skill.toml= manifest that declares what it does and
what capabilities it needs.  You should review this file before installing any
skill.

| Field                       | Description                                          |
|-----------------------------+------------------------------------------------------|
| =[skill] name=              | Unique identifier for the skill                      |
| =[skill] version=           | Version string                                       |
| =[skill] description=       | What the skill does                                  |
| =[skill] signature=         | Ed25519 signature (empty string = unsigned/dev only) |
| =[capabilities] http_domains= | Domains the skill may contact (empty = no network) |
| =[capabilities] read_files= | File paths the skill may read (empty = none)         |
| =[capabilities] write_files= | File paths the skill may write (empty = none)       |
| =[capabilities] env_vars=   | Environment variables the skill may see (empty = none)|
| =max_memory_mb=             | Maximum memory in megabytes                          |
| =max_cpu_seconds=           | Maximum CPU time per invocation                      |
| =[provides] functions=      | List of exported function names                      |

A skill with =http_domains = []= and =read_files = []= and =write_files = []=
is a pure computation skill — it can only process the data you give it.

Every WASM skill automatically has access to a small persistent key-value store
with no manifest declaration required.  The store is per-skill and age-encrypted
at rest (=~/.crichton/kv/<skill-id>.age=).  Quota limits apply (see =[kv]= in the
configuration reference).

** Managing WASM Skill Storage

Every WASM skill gets its own age-encrypted key-value store at
=~/.crichton/kv/<skill-id>.age=.  Most users never need to touch these files
directly, but the following operations are available via the Lisp REPL (SLIME/SLY
or =crichton-client --eval=).

*** Viewing storage usage

#+begin_example
How much storage are my WASM skills using?
#+end_example

Or from the REPL:

#+begin_src lisp
(crichton/skills:kv-usage-report)
#+end_src

*** Health check

Checks that all store files are accessible and not corrupt:

#+begin_src lisp
(crichton/skills:kv-health-check)
;; KV Store Health Check
;; ----------------------------------------
;;   Directory  : /home/you/.crichton/kv/
;;   Accessible : yes
;;   Files      : 2 (2 OK, 0 corrupt)
;; ...
#+end_src

Returns =T= when everything is healthy.

*** Backup

Copy all skill stores to a backup directory (the daemon flushes any pending
writes first):

#+begin_src lisp
(crichton/skills:backup-kv-store #p"~/.crichton/kv-backup-2026-03-22/")
#+end_src

Or from the shell:

#+begin_src bash
# Simple file copy — .age files are already encrypted
cp -r ~/.crichton/kv /secure/backup/kv-$(date +%Y%m%d)
#+end_src

*** Restore

#+begin_src lisp
(crichton/skills:restore-kv-backup #p"~/.crichton/kv-backup-2026-03-22/")
#+end_src

This copies the backed-up =.age= files back into =~/.crichton/kv/= and clears the
in-memory cache so they are reloaded on next access.

*** Integrity check and repair

If you suspect a skill's store is corrupt (e.g., interrupted write, damaged
backup):

#+begin_src lisp
;; Find corrupt files
(crichton/skills:check-kv-integrity)
;; => ((:SKILL-ID "my-skill" :STATUS :OK :ERROR NIL)
;;     (:SKILL-ID "bad-skill" :STATUS :CORRUPT :ERROR "..."))

;; Rename the corrupt file out of the way (adds .corrupt extension)
;; and evict the cache entry so the skill starts fresh
(crichton/skills:repair-corrupt-kv "bad-skill")
#+end_src

*** Clearing a skill's data

#+begin_src lisp
;; Wipe all stored data for one skill
(crichton/skills:kv-clear-skill "my-skill")
#+end_src

This removes the =.age= file and the in-memory cache entry.

*** Quota limits

Default limits per skill (adjustable in =config.toml= under =[kv]=):

| Limit             | Default    |
|-------------------+------------|
| Max keys          | 100        |
| Max value size    | 10 KiB     |
| Max total storage | 1 MiB      |

When a skill exceeds a quota, the =kv_set= call returns an error and the skill
receives a −1 return code.  The daemon logs a warning.  Increase limits in
=config.toml= if a skill legitimately needs more space.

* Troubleshooting

** Daemon will not start

Check the journal for error messages:

#+begin_src bash
journalctl --user -u crichton -n 50
#+end_src

Common causes:

- *SBCL not found* — make sure SBCL is installed and =/usr/bin/sbcl= exists.
  If it is in a different location, edit the service file.
- *Wrong path in service file* — double-check the path to your crichton source
  directory in =~/.config/systemd/user/crichton.service=.
- *Missing dependency* — SBCL will print what it could not load.  You may need
  to install the missing ASDF package via Quicklisp.

** "Daemon socket not found"

The socket =~/.crichton/daemon.sock= is created when the daemon starts and
removed when it stops.  This error means the daemon is not running:

#+begin_src bash
systemctl --user status crichton
systemctl --user start crichton
#+end_src

** "Credential not found"

The credential was not stored, or was stored under a different name.  List
what is stored:

#+begin_src bash
ls ~/.crichton/credentials/
#+end_src

Each file =<name>.age= corresponds to a credential named =<name>=.  Store the
missing credential as described in the relevant Built-in Skills section.

** Weather returns no results

The weather skill covers Canadian cities only (Environment Canada data).  For
other countries, there is no built-in weather source yet.

Also check that =api.weather.gc.ca= is in your =allowed_builtin_http_domains=
if you have =egress_policy = "deny-all"=.

** TUI is stuck on "Streaming..."

This can happen if the daemon encounters an error during a streaming response.
Press Ctrl-C once (then a second time to quit if needed), and restart the TUI.
Check the logs:

#+begin_src bash
journalctl --user -u crichton -n 20
#+end_src

** Rate limit errors from the Anthropic API

If you see errors mentioning HTTP 429, you are hitting Anthropic's rate limits.
Crichton automatically retries up to three times with 15/30/45-second backoff,
so usually the request will succeed on retry.  If it keeps failing, you may
need to upgrade your Anthropic API tier.

** Org-mode skill reports "Org-mode skill is disabled"

You need to set =enable = true= in the =[orgmode]= section of
=~/.crichton/config.toml=, and also set at least one path in =allowed_paths=.
An empty =allowed_paths= list denies all file access even if the skill is
enabled.

** Checking the logs

All daemon logs are written to =~/.crichton/logs/daemon.jsonl= in JSON Lines
format.  You can ask Crichton to inspect them for you:

#+begin_example
Show me the last 30 log entries.
Search the logs for any errors from the last hour.
#+end_example

Or use standard tools:

#+begin_src bash
# Tail the raw log
tail -f ~/.crichton/logs/daemon.jsonl

# Pretty-print recent entries (requires jq)
tail -50 ~/.crichton/logs/daemon.jsonl | jq .

# Filter for errors
grep '"level":"error"' ~/.crichton/logs/daemon.jsonl | tail -20
#+end_src
