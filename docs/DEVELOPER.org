* Introduction

Crichton is a secure background AI agent daemon built in SBCL Common Lisp.
/Named for "The Admirable Crichton" --- the quietly competent servant who's
actually running everything./  This document provides comprehensive technical
documentation for developers working on or integrating with Crichton.

**Design Philosophy**: Security-first microkernel architecture informed by a
real-world audit of OpenClaw.  All external code runs in WASM sandboxes with
capability-based permissions.

**Target Audience**: System developers, skill authors, integrators, and
contributors to the Crichton codebase.

* Design Principles

Derived from OpenClaw audit findings:

1. *Skills must be isolated* --- no same-process execution (OpenClaw: CRITICAL finding)
2. *Credentials must be encrypted at rest* --- not just file permissions (OpenClaw: plaintext with 0600)
3. *Network egress must be controlled* --- skills cannot phone home (OpenClaw: unrestricted)
4. *Code must be signed* --- no unsigned skill execution (OpenClaw: none)
5. *Defense in depth* --- no single trust boundary (OpenClaw: "disk access is the trust boundary")

** Language/Runtime
The daemon, skill runner, and client interface are all written in Common Lisp
(specifically SBCL).

*Rationale*:
- *Live-coding*: Connect SLIME/SLY to the running daemon, redefine functions,
  inspect state, fix bugs without restart.  This is the killer feature for a
  long-running agent daemon.
- *Condition system*: Restartable errors --- the daemon can encounter problems and
  offer recovery strategies rather than crashing.
- *Performance*: SBCL compiles to native code.  Competitive with Go for most
  workloads.
- *CFFI*: Mature foreign function interface for calling C libraries (wasmtime, OS
  keychains, libsecret, etc.)

*WASM integration path*: SBCL CFFI \rightarrow wasmtime C API (or wasmer C API).  No existing
CL bindings exist --- we'll build thin CFFI wrappers as part of the project.  This
is standard CL practice.

*Key CL libraries to leverage*:
- =cffi= --- FFI to wasmtime, libsecret, OS keychain APIs
- =bordeaux-threads= --- portable threading
- =usocket= / =sb-bsd-sockets= --- networking
- =ironclad= --- pure CL crypto (Ed25519 signing, age encryption)
- =cl-json= or =shasht= --- JSON handling
- =hunchentoot= or =clack= --- HTTP server (for webhook endpoints)
- =log4cl= --- structured logging
- =dexador= --- HTTP client (for LLM API calls)

** Components

*** Daemon (~crichton~ executable)
- *Purpose*: Main agent process, LLM integration, credential management
- *Runs as*: systemd user service
- *Listens on*: =~/.crichton/daemon.sock= (Unix domain socket)
- *Protocol*: NDJSON (newline-delimited JSON)
- *Lifecycle*: Starts fresh on boot (no save-lisp-and-die), long-running

*** Client (~crichton-client~ binary)
- *Purpose*: Minimal command-line interface for interacting with daemon
- *Dependencies*: Minimal (shasht, sb-bsd-sockets, cl-readline)
- *No dependencies on*: cl+ssl, wasmtime, heavy libraries
- *Protocol*: NDJSON over =daemon.sock=

*** TUI Client (~crichton-tui~ binary)
- *Purpose*: Interactive terminal UI with colour, layout, streaming, and push notifications
- *Architecture*: TEA (The Elm Architecture) via cl-tuition
- *Dependencies*: tuition (cl-tuition), shasht, bordeaux-threads, cl-ppcre
- *No dependencies on*: cl+ssl, wasmtime, ncurses
- *Protocol*: NDJSON over =daemon.sock= (same as crichton-client, with =subscribe= and =stream= extensions)
- *Lifecycle*: Standalone binary via =save-lisp-and-die=; also supports one-shot CLI mode
- *Features*:
  - Full ANSI colour (256-colour and truecolour)
  - Adaptive layout with terminal resize handling (=SIGWINCH=)
  - Markdown-to-ANSI rendering of LLM responses
  - Real-time streaming of LLM responses (=chat_delta= / =chat_done= push messages)
  - Push notification toasts from daemon (battery, RSS, scheduler alerts)
  - Scrollable chat viewport with page up/down, home/end
  - Input editing with cursor support (=tui:text-input= component)
  - Spinner animation during LLM thinking
  - Sixel graphics infrastructure (detection, =img2sixel= integration)
  - Local commands: =:quit=, =:clear=, =:new=, =:session=, =:status=, =:notifications=, =:sixel on/off=, =:img=

*** Emacs Client (~emacs/crichton.el~)
- *Purpose*: Native Emacs interface for chatting with the daemon
- *Dependencies*: Emacs 28.1+ (built-in =json=, =comint=, =cl-lib= only)
- *Protocol*: NDJSON over =daemon.sock= (same as other clients)
- *Features*:
  - Comint-based chat buffer (=*crichton-chat*=) with input history
  - Hidden process buffer (=*crichton-process*=) captures raw LLM streaming deltas, like SLIME's =*inferior-lisp*=
  - =[thinking…]= overlay while LLM is working; final response appears on =chat_done=
  - =C-c C-p=: peek at raw stream; =C-c C-o=: pop last response into org-mode buffer; =C-c C-k=: disconnect
  - Bidirectional RPC: daemon can invoke allowlisted Emacs functions via =emacs_invoke= (org-roam queries, org-agenda, etc.)
  - Push notification display in chat buffer
- *Entry points*: =M-x crichton-connect=, =M-x crichton-chat=

*** Runner Process
- *Purpose*: Isolated WASM execution environment
- *Spawned by*: Daemon on-demand
- *Communication*: NDJSON over Unix socket
- *Lifecycle*: Short-lived per-invocation (currently)
- *Sandboxing*: WASM runtime (wasmtime) with capability manifests

*** WASM Skills
- *Language*: Any language compiling to WASM (Rust, C, AssemblyScript, etc.)
- *Execution*: Inside runner process via wasmtime
- *Capabilities*: Declared in =skill.toml=, enforced by runner
- *Host functions*: =log=, =kv_get=, =kv_set=, =http_fetch=, etc.

- *Trust Level 1* (User): Local system, file permissions, systemd
- *Trust Level 2* (Daemon): Credential access, LLM API keys, session management
- *Trust Level 3* (Runner): WASM execution, capability enforcement
- *Trust Level 4* (Skills): Untrusted code, capability-constrained

*Trust boundary*: The RPC interface between daemon and skill-runner.  Skills never
run in the daemon process.  The runner process can be killed/resource-limited
independently.

** Threading Model

*** Daemon
- *Main thread*: RPC accept loop
- *Worker threads*: Per-client connection handlers
- *Background threads*: Scheduler, channel managers (Discord, etc.)
- *Synchronization*: Bordeaux-threads locks for shared state

*** TUI Client
- *Main thread*: cl-tuition program loop (terminal I/O, rendering, input decoding)
- *Daemon I/O thread*: Persistent connection to =daemon.sock=; reads NDJSON lines and injects messages via =tui:send=
- *Commands*: Zero-arg lambdas run on cl-tuition's thread pool (e.g., socket writes for chat requests)
- *Invariant*: The daemon I/O thread never touches terminal state directly

*** Runner
- *Main thread*: RPC accept loop
- *Worker thread*: Per-invocation handler
- *WASM execution*: Synchronous, blocking

** Data Flow: Chat Request (Non-Streaming)

1. User sends message via client \rightarrow daemon socket
2. Daemon receives NDJSON request, extracts text and session_id
3. Daemon loads session history from =~/.crichton/sessions/=
4. Daemon invokes LLM provider (Anthropic Claude)
5. LLM may request tool use (e.g., weather, system-info)
6. Daemon executes tools (built-in or WASM skills)
7. For WASM skills: daemon \rightarrow runner \rightarrow wasmtime \rightarrow skill
8. Results flow back: skill \rightarrow wasmtime \rightarrow runner \rightarrow daemon
9. Daemon sends tool results to LLM
10. LLM generates final response
11. Daemon saves session history
12. Daemon returns response to client
13. Client displays to user

** Data Flow: Chat Request (Streaming)

When the client sends ={"stream": true}= in the chat request, the daemon streams
the LLM response incrementally:

1. TUI sends ={op: "chat", text: "...", stream: true}= \rightarrow daemon socket
2. Daemon receives request, sets up session (same as non-streaming)
3. Daemon calls =run-agent/stream= with an =on-delta= callback
4. Agent loop iterates:
   a. Calls =stream-message= on the LLM provider (Anthropic SSE)
   b. If LLM responds with tool use: execute tools, loop again (no deltas pushed)
   c. If LLM responds with text: SSE =text_delta= events fire the =on-delta= callback
5. For each text delta: daemon pushes ={op: "chat_delta", id: N, text: "..."}= to client
6. On completion: daemon pushes ={op: "chat_done", id: N, text: "...", session_id: "..."}=
7. Daemon also returns a standard OK response (backward compatibility)
8. TUI renders deltas incrementally into a placeholder chat entry
9. TUI ignores the duplicate OK response via =handled-stream-ids= tracking

*Key properties*:
- Streaming uses the same per-connection =(stream . lock)= pattern as push notifications
- Tool-use iterations are invisible to the client --- only the final text response streams.
  During tool loops, =run-agent/stream= calls the non-streaming =send-message= and executes
  tools; only when the LLM produces a final text response does streaming resume.
- Graceful fallback: if the provider does not support streaming, =run-agent/stream= falls back to =send-message= automatically
- Session snapshot/rollback: =handle-streaming-chat-request= takes a shallow copy of
  session messages before calling =run-agent/stream=.  An =unwind-protect= restores the
  snapshot if the call errors or the client disconnects mid-stream.  This prevents torn
  session state (e.g., a =tool_use= block appended to history without a matching
  =tool_result=).  =run-agent-loop= =copy-list=s its input on entry for the same reason.

*** Concurrent Session Claim Contention

Sessions are protected by a claim token so that two concurrent requests
cannot corrupt the same session's message history.  A second request
arriving while the first is still running gets an immediate =session-busy=
error rather than racing.

* Wire Formats & Protocols

** NDJSON Protocol Overview

All inter-process communication uses **NDJSON** (Newline-Delimited JSON) over Unix domain sockets.

- *Format*: One JSON object per line
- *Encoding*: UTF-8
- *Size limit*: 8 MB per message (configurable via =*max-message-bytes*=)
- *Transport*: Unix domain sockets (no network exposure)

** Daemon RPC Protocol (daemon.sock)

*** Request Format

#+begin_src json
{
  "id": 123,
  "op": "operation_name",
  "field1": "value1",
  "field2": "value2"
}
#+end_src

- =id=: Integer correlation ID (unique per request)
- =op=: Operation string (e.g., "chat", "status", "ping")
- Additional fields depend on operation

*** Response Format (Success)

#+begin_src json
{
  "id": 123,
  "ok": true,
  "result": { ... }
}
#+end_src

- =id=: Matches request ID
- =ok=: Boolean success flag (true)
- =result=: Operation-specific result (any JSON type)

*** Response Format (Error)

#+begin_src json
{
  "id": 123,
  "ok": false,
  "error": {
    "code": "error_code",
    "message": "Human-readable error message"
  }
}
#+end_src

- =id=: Matches request ID
- =ok=: Boolean success flag (false)
- =error.code=: Machine-readable error code
- =error.message=: Human-readable error description

*** =chat=

Send a message to the agent.

*Request*:
#+begin_src json
{
  "id": 2,
  "op": "chat",
  "text": "What's the weather like?",
  "session_id": "sess_abc123",
  "stream": true
}
#+end_src

- =text=: User message (required)
- =session_id=: Session identifier (optional, auto-generated if omitted)
- =stream=: Enable streaming response (optional, default false)

*Response* (non-streaming, or final response after streaming):
#+begin_src json
{
  "id": 2,
  "ok": true,
  "result": {
    "text": "The current weather is...",
    "session_id": "sess_abc123"
  }
}
#+end_src

*Streaming push messages* (when =stream= is true):

The daemon pushes incremental text deltas to the client connection before
sending the final response.  These are unsolicited messages (no =ok= field):

#+begin_src json
{"op": "chat_delta", "id": 2, "text": "The current"}
{"op": "chat_delta", "id": 2, "text": " weather is"}
{"op": "chat_delta", "id": 2, "text": " sunny."}
{"op": "chat_done", "id": 2, "text": "The current weather is sunny.", "session_id": "sess_abc123"}
#+end_src

- =chat_delta=: Incremental text fragment (id matches the original request)
- =chat_done=: Final complete text with session ID; signals streaming completion

Implementation: =daemon/rpc-server.lisp= =handle-streaming-chat-request=, uses
=run-agent/stream= from =agent/loop.lisp=.

*Session claim mechanism*: Each incoming chat request must acquire a
per-session claim before entering the agent loop.  This prevents two
concurrent requests from corrupting the same session's message history.
If a claim is already held (i.e., a previous request is still running),
the daemon returns an error response with =result: "session-busy"=.

Claims use a token-based release: each claim is a =(token . claimed-at)=
pair.  The handling thread releases the claim by token; a stale thread
that wakes up after a timeout has already replaced its claim is a no-op.
Claims are force-released after =*chat-session-claim-timeout*= seconds
(default 60) to recover from aborted requests.  See
=session-claim= / =session-release= in =daemon/rpc-server.lisp=.

*** =subscribe=

Register the current connection for push notifications.

*Request*:
#+begin_src json
{
  "id": 5,
  "op": "subscribe"
}
#+end_src

*Response*:
#+begin_src json
{
  "id": 5,
  "ok": true,
  "result": true
}
#+end_src

After subscribing, the daemon will push unsolicited notification messages to
the connection whenever skills or monitors generate events:

#+begin_src json

{"op": "notify", "kind": "rss", "text": "New item: ...", "source": "rss-monitor"}
{"op": "notify", "kind": "scheduler", "text": "Task completed", "source": "scheduler"}
#+end_src

- =kind=: Notification category (string)
- =text=: Human-readable message (string)
- =source=: Originating skill or monitor (string)

Implementation: =daemon/notifications.lisp= manages a subscriber registry of
=(stream . lock)= pairs.  =notification-post= iterates subscribers and writes
with per-stream locks held.  Dead subscribers are automatically removed.

** Runner RPC Protocol

Communication between daemon and runner process.

*** =invoke=

Execute a WASM module.

*Request*:
#+begin_src json
{
  "id": 10,
  "op": "invoke",
  "wasm_b64": "<base64-encoded WASM bytes>",

- =wasm_b64=: Base64-encoded WASM binary (mutually exclusive with =wat=)

- =nresults=: Number of expected return values (default: 1)
- =manifest=: Capability manifest plist (optional)

*Response*:
#+begin_src json
{
  "id": 10,
  "ok": true,
  "result": [123]
}
#+end_src

*** JSON-Through-Memory ABI

For complex data exchange (used by =invoke_json=):

1. Skill exports: =alloc(size: i32) -> i32=
2. Skill exports: =dealloc(ptr: i32)=
3. Runner serializes input JSON to bytes
4. Runner calls =alloc(size)= to get WASM memory pointer
5. Runner writes JSON bytes to WASM memory at pointer
6. Runner calls skill's main export (e.g., =filter_items(ptr, len)=)
7. Skill returns =(ptr, len)= pair (packed as single i64)
8. Runner reads output JSON from WASM memory
9. Runner calls =dealloc(ptr)=

Rust example:
#+begin_src rust
#[no_mangle]
pub extern "C" fn alloc(size: usize) -> *mut u8 {
    let mut buf = Vec::with_capacity(size);
    let ptr = buf.as_mut_ptr();
    std::mem::forget(buf);
    ptr
}

#[no_mangle]
pub extern "C" fn dealloc(ptr: *mut u8) {
    unsafe { drop(Vec::from_raw_parts(ptr, 0, 0)); }
}

#[no_mangle]
pub extern "C" fn main_function(input_ptr: *const u8, input_len: usize) -> i64 {
    // Parse JSON from input
    let input_json = unsafe {
        std::slice::from_raw_parts(input_ptr, input_len)
    };

    // Process and create output JSON
    let output_bytes = b"{ \"result\": 123 }";
    let output_ptr = alloc(output_bytes.len());
    unsafe {
        std::ptr::copy_nonoverlapping(
            output_bytes.as_ptr(),
            output_ptr,
            output_bytes.len()
        );
    }

    // Return (ptr, len) as packed i64
    ((output_ptr as u64) << 32) | (output_bytes.len() as u64)
}
#+end_src

** KV Store API (~skills/kv-store.lisp~) (cricht-jio)

Per-skill age-encrypted persistent key-value store for WASM skills.

*** Overview

- *Location*: =~/.crichton/kv/<skill-id>.age= — one file per skill
- *Format*: Age-encrypted JSON envelope (=version=, =skill_id=, =updated_at=, =data=)
- *Isolation*: Each skill can only access its own KV store (enforced at host function level)
- *Thread safety*: All operations protected by =*kv-lock*= (bordeaux-threads)
- *Write-through cache*: writes go to both in-memory hash-table and disk atomically

*** Quotas (configurable in =config.toml= under =[kv]=)

| Parameter         | Default  | Description                          |
|-------------------+----------+--------------------------------------|
| =max_keys=        | 100      | Max keys per skill                   |
| =max_value_bytes= | 10240    | Max value size in bytes (10 KiB)     |
| =max_total_bytes= | 1048576  | Max total store size per skill (1 MiB) |

** Retry and Timeout Infrastructure (~skills/retry-infrastructure.lisp~)

Shared retry, backoff, and timeout utilities used across skill modules
and the agent loop.

*** =with-retry= Macro

#+begin_src lisp
(with-retry (:context :weather :max-retries 3)
  (dex:get "https://api.weather.gc.ca/..."))
#+end_src

Wraps BODY in a retry loop with exponential backoff.  Options:

| Option          | Default                    | Description                                      |
|-----------------+----------------------------+--------------------------------------------------|
| =:max-retries=  | =*default-max-retries*= (3) | Maximum retry attempts                           |
| =:backoff-base= | 1 second                   | Base delay; doubles each attempt                 |
| =:max-backoff=  | 30 seconds                 | Cap on per-attempt delay                         |
| =:context=      | =nil=                      | Symbol for context-specific =transient-error-p= |
| =:on-retry=     | =nil=                      | =(lambda (condition attempt delay))= hook        |

Error classification via =transient-error-p= generic: default method
retries on socket errors, 5xx/429/503 HTTP status, and connection/timeout
error messages.  Context-specific methods (=:weather=, =:rss=, =:github=)
refine this (e.g., weather 404 is never retried — city not found).

*** =with-timeout= Macro

#+begin_src lisp
(with-timeout (30 :error-message "Weather API timed out")
  (fetch-weather city))
#+end_src

Executes BODY with a watchdog thread.  If BODY does not complete within
SECONDS, the watchdog fires =bt:interrupt-thread= into the calling thread
and signals =operation-cancelled=.

Used for tool execution and non-streaming LLM API calls.  *Not* used for
SSE streams — see the SSE Idle Watchdog section below.

*** SSE Idle Watchdog

Streaming LLM responses use a separate, fd-close based watchdog rather
than =with-timeout=.  The reason: =bt:interrupt-thread= is unreliable for
threads blocked inside OpenSSL's =SSL_read=, which ignores POSIX signals.
Closing the underlying socket fd unconditionally unblocks any pending read.

The watchdog is implemented in =process-sse-stream= (=llm/anthropic.lisp=):

1. A watchdog thread starts alongside the SSE reader.
2. If no bytes arrive for =*sse-stream-timeout*= seconds (default 120,
   configurable via =[llm] sse-stream-timeout= in =config.toml=), the
   watchdog calls =(close response-stream :abort t)=.
3. The fd close unblocks the SSL read; the resulting error is caught by
   the SSE reader, which checks a =watchdog-fired= flag and re-signals
   it as =operation-cancelled=.
4. A =stream-done= flag prevents the watchdog from firing after normal
   stream completion.

This is an /idle/ timeout — it resets on every chunk received.  The outer
=*llm-api-timeout*= (300 s) is the /elapsed/ ceiling for the entire call.

*** =operation-cancelled= Condition

=operation-cancelled= is a distinct subclass of =error= signalled by
=with-timeout= and by the SSE idle watchdog.  Callers that catch broad
=(error (c) ...)= conditions *must* re-raise it rather than swallowing it:

#+begin_src lisp
(handler-case
    (handle-sse-event ...)
  (crichton/skills:operation-cancelled (c) (error c))  ; re-raise
  (error (c)
    (log:warn "SSE event error: ~A" c)))                ; swallow
#+end_src

This pattern appears in =dispatch-sse-event= so that a timeout interrupt
delivered during JSON parsing of an SSE event propagates correctly rather
than being silently discarded.

*** HTTP helpers

=http-get-with-retry=, =http-post-with-retry=, =http-put-with-retry=,
=http-delete-with-retry= — convenience wrappers around =dex:get= etc.
that apply =with-retry= automatically.

*** Agent Loop Timeouts

Three timeouts govern streaming LLM calls (configured in =config.toml=
under =[llm]=):

| Config key (=[llm]=)     | Default | Description                                                              |
|--------------------------+---------+--------------------------------------------------------------------------|
| =api-timeout=            | 300 s   | Elapsed ceiling for the entire LLM API call. Thread-interrupt watchdog.  |
|                          |         | Lisp: =*llm-api-timeout*=                                                |
| =tool-timeout=           | 60 s    | All tool calls in one agent iteration. Thread-interrupt watchdog.        |
|                          |         | Lisp: =*tool-execution-timeout*=                                         |
| =sse-stream-timeout=     | 120 s   | Idle silence on an SSE stream before fd-close. See SSE Idle Watchdog.   |
|                          |         | Lisp: =*sse-stream-timeout*=                                             |

The SSE idle timeout fires first on a stalled stream; the api-timeout is
the fallback if the stream hangs in a way the idle watchdog cannot detect.
All three signal =operation-cancelled= on expiry.

** LLM Provider Protocol (~llm/protocol.lisp~)

CLOS-based provider abstraction.

*** Classes

**** =llm-provider=

Base class for LLM providers.

Slots:
- =provider-id=: Keyword identifier (e.g., =:anthropic=)
- =model=: Model name string (e.g., ="claude-sonnet-4-20250514"=)

Generics to implement:
- =send-message (provider messages &key system max-tokens temperature tools)=
- =stream-message (provider messages on-event &key system max-tokens temperature tools)=

=stream-message= has a default method that signals =llm-feature-not-supported=.
=run-agent/stream= catches that condition and falls back to =send-message=
automatically — providers do not need to implement streaming to be usable.

*** Usage Metering Hooks

=llm/protocol.lisp= defines =:around= methods on both =send-message= and
=stream-message= that record token usage after every call via
=crichton/skills:record-usage=.

To avoid blocking the LLM call path, the storage write is performed in a
background thread via =%spawn-usage-persist=:

#+begin_src lisp
(bt:make-thread
  (lambda ()
    (handler-case
        (crichton/skills:record-usage "llm" model input-tokens output-tokens)
      (error (c) (log:warn "Failed to record LLM usage: ~A" c))))
  :name "llm-usage-persist")
#+end_src

The reason for the background thread: =record-usage= calls =persist-meter=
which acquires =*storage-lock*=.  The daemon's storage-sync task (scheduled
every 60 seconds) calls =flush-all-storage=, which holds =*storage-lock*=
for the duration of each =age-encrypt= subprocess call.  Running the persist
inline would have stalled the LLM response path on that lock contention.

*** Condition Types

- =llm-error=: Base error class
- =llm-api-error=: HTTP API errors (includes status code)
- =llm-auth-error=: Authentication failures
- =llm-rate-limit-error=: Rate limiting (includes =retry-after= slot)
- =llm-feature-not-supported=: Provider does not implement the requested feature

Discover, load, and invoke WASM skills.

Scan =~/.crichton/skills/= for skill manifests and populate registry.

#+begin_src lisp

;; => 3 (number of skills found)
#+end_src

**** =list-skills ()=

(list-skills)

;; => (:name "rss-filter" :version "0.1.0" :manifest ...)

**** =load-skill (name)=

Load a skill's WASM bytecode into memory.

#+begin_src lisp

;; => #<SKILL-ENTRY "rss-filter">
#+end_src

**** =invoke-skill (name &key entry-point params abi)=

Load and invoke a WASM skill.  Supports two calling conventions:

- *i32 ABI* (legacy): calls the export with no arguments, returns an integer.
- *JSON ABI*: serialises =:params= into WASM shared memory, calls the export
  with the JSON-through-memory protocol (=alloc=, =dealloc=, 4-arg signature),
  and returns a parsed Lisp structure.

Keyword arguments:

- =:entry-point= (string) --- Override the default export declared in the manifest.
- =:params= (plist, hash-table, or any JSON-serialisable Lisp structure) ---
  Data to pass to the skill.  When non-nil, the JSON ABI is selected
  automatically.
- =:abi= --- =:auto= (default), =:json=, or =:i32=.  Under =:auto=, JSON ABI
  is selected when =:params= is non-nil /or/ the manifest declares
  =abi = "json"= for the entry-point; otherwise i32.

An error is signalled if =:params= is provided but the effective ABI is =:i32=.
Input size is capped at =*max-skill-json-input-bytes*= (default 4 MB).

#+begin_src lisp
;; Legacy i32 skill (no params)
(invoke-skill "meaning-of-life")
;; => 42

;; JSON ABI skill with parameters
(invoke-skill "rss-filter"
  :entry-point "filter_items"
  :params (list :items items
                :keywords '("Carney" "Bank of Canada")
                :match-mode "any"))
;; => #<hash-table with :matches and :statistics>

;; Force JSON ABI even without params (sends {})
(invoke-skill "my-skill" :abi :json)
#+end_src

**** =unload-skill (name)=

Unload a skill from memory.

#+begin_src lisp
(unload-skill "rss-filter")
;; => T
#+end_src

** Skill Pipeline Executor (~skills/pipeline.lisp~)

Chain multiple skill invocations into a linear pipeline where later steps
can reference earlier results.

*** Core Functions

**** =execute-pipeline (steps)=

Execute an ordered list of steps.  Each step is a hash-table (from JSON) or
plist with:

| Key           | Type   | Description                                  |
|---------------+--------+----------------------------------------------|
| =id=          | string | Unique step identifier (required)            |
| =kind=        | string | ="wasm"=, ="builtin"=, or ="auto"= (default) |
| =skill=       | string | Skill name (for WASM steps)                  |
| =entry_point= | string | WASM export or builtin name                  |
| =params=      | object | Input parameters (may contain refs)          |
| =builtin=     | string | Built-in function name (for builtin steps)   |

Returns a hash-table of step-id → result.

#+begin_src lisp
;; Two-step pipeline: fetch RSS, then filter with WASM skill
(execute-pipeline
  (list
    ;; Step 1: fetch RSS feed (built-in)
    (alexandria:plist-hash-table
      '("id" "fetch" "builtin" "rss_fetch"
        "params" #.(alexandria:plist-hash-table
                     '("url" "https://example.com/rss") :test 'equal))
      :test 'equal)
    ;; Step 2: filter items (WASM) — references step 1 output
    (alexandria:plist-hash-table
      '("id" "filter" "skill" "rss-filter"
        "entry_point" "filter_items"
        "params" #.(alexandria:plist-hash-table
                     '("items" #.(alexandria:plist-hash-table
                                   '("ref" "fetch.items") :test 'equal)
                       "keywords" #("AI" "LLM")
                       "match_mode" "any") :test 'equal))
      :test 'equal)))
;; => #<hash-table "fetch" => ..., "filter" => ...>
#+end_src

*** Reference Resolution

References use JSON objects with a ="ref"= key containing a dotted path:

#+begin_src json
{"ref": "step_id.key.subkey"}
#+end_src

Path segments navigate into:
- Hash-tables (string keys)
- Plists (keyword keys, with underscore-to-hyphen conversion)
- Vectors/lists (numeric index, e.g. ="items.0.title"=)

*** Built-in Pipeline Functions

Register built-in Lisp functions as pipeline steps:

#+begin_src lisp
(register-pipeline-builtin "my_func"
  (lambda (params)
    (do-something (gethash "input" params))))
#+end_src

Default builtins (registered via =register-default-pipeline-builtins=):

| Name         | Params                              | Description        |
|--------------+-------------------------------------+--------------------|
| =rss_fetch=  | =url= (string), =max_items= (int)  | Fetch RSS feed     |
| =rss_check=  | =url= (string), =max_items= (int)  | Check for new items|
| =weather=    | =city= (string)                     | Weather conditions |

*** Error Handling

=pipeline-error= condition carries =step-id= and =step-skill= slots:

#+begin_src lisp
(handler-case (execute-pipeline steps)
  (pipeline-error (c)
    (format t "Failed at step ~A: ~A"
            (pipeline-error-step-id c) c)))
#+end_src

** RSS Data Protocols

The built-in RSS reader (~skills/builtins/rss.lisp~) and the =rss-filter=
WASM skill (~examples/skills/rss-filter/~) communicate through a shared JSON
schema.  This section documents the exact data formats produced and consumed
at each stage.

*** Canonical RSS Item Schema

All RSS functions produce items using a single canonical plist schema (via
=feed-item-to-plist=).  When serialized to JSON by =normalize-for-json= in
=wasm/engine.lisp=, plist keywords become lowercase underscored strings
(e.g. =:feed-name= → =feed_name=).

*Plist form (Lisp side):*

#+begin_src lisp
(:id "https://example.com/article-1"   ; guid or link (required)
 :title "Article Title"                 ; item title (required)
 :description "Summary text..."         ; item summary
 :link "https://example.com/article-1"  ; URL to full article
 :published "Mon, 10 Feb 2026 12:00:00" ; publication date string
 :feed-name "Example Feed")             ; source feed title
#+end_src

*JSON form (WASM side):*

#+begin_src json
{
  "id": "https://example.com/article-1",
  "title": "Article Title",
  "description": "Summary text...",
  "link": "https://example.com/article-1",
  "published": "Mon, 10 Feb 2026 12:00:00",
  "feed_name": "Example Feed"
}
#+end_src

| Field         | Plist key      | JSON key      | Type   | Required | Notes                             |
|---------------+----------------+---------------+--------+----------+-----------------------------------|
| Identifier    | =:id=          | =id=          | string | yes      | From RSS =<guid>= or =<link>=    |
| Title         | =:title=       | =title=       | string | yes      |                                   |
| Description   | =:description= | =description= | string | no       | RSS =<description>= / Atom =<summary>= |
| Content       | (not produced) | =content=     | string | no       | Reserved for full-text (future)   |
| Link          | =:link=        | =link=        | string | no       | URL to the full article           |
| Published     | =:published=   | =published=   | string | no       | Date string (format varies by feed) |
| Feed name     | =:feed-name=   | =feed_name=   | string | no       | Title of the source feed          |

*** =rss_fetch= Output (Built-in)

=rss-fetch= returns a plist describing the feed and its items:

#+begin_src lisp
(rss-fetch "https://example.com/rss" :max-items 50)
;; =>
(:url "https://example.com/rss"
 :title "Example Feed"
 :format :rss            ; :rss, :rdf, or :atom
 :count 12
 :items ((:id "..." :title "..." :description "..." :link "..."
          :published "..." :feed-name "Example Feed")
         ...))
#+end_src

Serialized to JSON (e.g. as pipeline step output):

#+begin_src json
{
  "url": "https://example.com/rss",
  "title": "Example Feed",
  "format": "rss",
  "count": 12,
  "items": [
    {
      "id": "...",
      "title": "...",
      "description": "...",
      "link": "...",
      "published": "...",
      "feed_name": "Example Feed"
    }
  ]
}
#+end_src

*** =rss_check= Output (Built-in)

=rss-check= returns only new (unseen) items:

#+begin_src lisp
(rss-check "https://example.com/rss")
;; =>
(:url "https://example.com/rss"
 :title "Example Feed"
 :new-count 3
 :total-count 12
 :new-items ((:id "..." :title "..." :description "..." :link "..."
              :published "..." :feed-name "Example Feed")
             ...))
#+end_src

*** =rss-filter= / =filter_items= Input (WASM)

The =filter_items= function expects a JSON object containing the items to
filter and keyword parameters:

#+begin_src json
{
  "items": [                          
    {
      "id": "guid-1",
      "title": "Carney announces rate decision",
      "description": "Bank of Canada governor spoke today",
      "link": "https://example.com/1",
      "published": "2026-02-10T12:00:00Z",
      "feed_name": "CBC News"
    }
  ],
  "keywords": ["Carney", "Bank of Canada"],
  "match_mode": "any",
  "search_fields": ["title", "description"],
  "case_sensitive": false
}
#+end_src

| Field            | Type          | Required | Default                    | Description                    |
|------------------+---------------+----------+----------------------------+--------------------------------|
| =items=          | array         | yes      | —                          | RSS items (canonical schema)   |
| =keywords=       | array         | yes      | —                          | Strings to match against items |
| =match_mode=     | string        | no       | ="any"=                    | ="any"= (OR) or ="all"= (AND) |
| =search_fields=  | array         | no       | =["title","description"]=  | Fields to search               |
| =case_sensitive= | boolean       | no       | =false=                    | Case-sensitive matching        |

*** =rss-filter= / =filter_items= Output (WASM)

Returns matched items with the keywords that triggered each match:

#+begin_src json
{
  "matches": [
    {
      "id": "guid-1",
      "title": "Carney announces rate decision",
      "description": "Bank of Canada governor spoke today",
      "link": "https://example.com/1",
      "published": "2026-02-10T12:00:00Z",
      "feed_name": "CBC News",
      "matched_keywords": ["Carney", "Bank of Canada"]
    }
  ],
  "statistics": {
    "items_scanned": 12,
    "items_matched": 1
  }
}
#+end_src

*** Pipeline Composition

The schemas are designed to compose via pipeline references.  The =items=
field from =rss_fetch= output feeds directly into the =items= parameter of
=filter_items=:

#+begin_src json
[
  {
    "id": "fetch",
    "builtin": "rss_fetch",
    "params": {"url": "https://www.cbc.ca/webfeed/rss/rss-topstories"}
  },
  {
    "id": "filter",
    "skill": "rss-filter",
    "entry_point": "filter_items",
    "params": {
      "items": {"ref": "fetch.items"},
      "keywords": ["Carney", "Bank of Canada"],
      "match_mode": "any"
    }
  }
]
#+end_src

Equivalent Lisp invocation (without pipelines):

#+begin_src lisp
(let* ((feed (rss-fetch "https://www.cbc.ca/webfeed/rss/rss-topstories"))
       (result (invoke-skill "rss-filter"
                 :entry-point "filter_items"
                 :params (list :items (getf feed :items)
                               :keywords '("Carney" "Bank of Canada")
                               :match-mode "any"))))
  (getf result :matches))
#+end_src

*** JSON Serialization Notes

Lisp plist params are automatically converted to JSON objects by
=normalize-for-json= in =wasm/engine.lisp= before WASM invocation:

- Keyword keys are downcased and hyphens become underscores:
  =:feed-name= → ="feed_name"=, =:match-mode= → ="match_mode"=
- Nested plists are recursively converted
- Hash-tables pass through unchanged
- Lists that don't look like plists (first element not a keyword)
  are serialized as JSON arrays

*** RSS Feed Writing API

Crichton can produce RSS 2.0 feeds in addition to consuming them.  The API
lives in =skills/builtins/rss.lisp= under the heading "Feed writing/generation".
Items are persisted to encrypted storage (namespace =rss-pub=) and survive
daemon restarts.

*Feed lifecycle*:

#+begin_src lisp
;; 1. Configure (optional — auto-created with defaults on first publish)
(rss-feed-configure "my-feed"
  :title "My Feed"
  :description "Updates from Crichton"
  :link "https://example.com"
  :max-items 50)

;; 2. Publish items
(rss-feed-publish "my-feed"
  :title "Something happened"
  :description "Details about the event."
  :link "https://example.com/post-1")
;; => "https://example.com/post-1#<timestamp>"   ; GUID

;; 3. Retrieve XML to serve
(rss-feed-xml "my-feed")
;; => "<?xml version=\"1.0\" encoding=\"UTF-8\"?>
;;     <rss version=\"2.0\">..."

;; 4. Inspect
(rss-feed-items "my-feed")
;; => ((:title "Something happened" :description "..." :link "..." :guid "..." :pub-date "...") ...)

(rss-feed-list)
;; => ("my-feed" ...)

;; 5. Housekeeping
(rss-feed-clear "my-feed")   ; remove items, keep config
(rss-feed-delete "my-feed")  ; remove config and items
#+end_src

*Function reference*:

| Function               | Arguments                                          | Returns          |
|------------------------+----------------------------------------------------+------------------|
| =rss-feed-configure=   | name &key title description link max-items         | feed name        |
| =rss-feed-publish=     | name &key title description link guid pub-date     | GUID string      |
| =rss-feed-items=       | name                                               | list of plists   |
| =rss-feed-xml=         | name                                               | XML string       |
| =rss-feed-list=        | —                                                  | list of names    |
| =rss-feed-clear=       | name                                               | T                |
| =rss-feed-delete=      | name                                               | T                |

*Storage details*:

- Namespace: =rss-pub= (file: =~/.crichton/data/rss-pub.age=)
- Config key: =feed-config:<name>=  (hash-table with "title", "description", "link", "max_items")
- Items key:  =feed-items:<name>=   (vector of hash-tables, newest first)
- Items are capped at =max_items= (default 100); oldest entries are dropped.
- Dates are stored in RFC 822 format ("Mon, 01 Jan 2026 00:00:00 +0000") as
  required by RSS 2.0 validators.
- XML output escapes =&=, =<=, =>= in all text fields.

*Agent tool actions* (in the =rss= tool):

=publish_item=, =configure_feed=, =get_feed_xml=, =list_feed_items=,
=list_feeds=, =clear_feed=, =delete_feed= (see =agent/tools.lisp=).

*** =define-tool= Macro

All built-in tools are defined with the =define-tool= macro, which
eliminates the boilerplate of =make-properties= + =register-tool= +
=make-json-schema= + handler lambda + =hget= extraction.

#+begin_src lisp
(define-tool weather
    (:description "Get current weather conditions and forecast.")
  ((city "string" "Canadian city name for weather lookup."))
  (with-output-to-string (s)
    (crichton/skills:weather-report :city city :stream s)))
#+end_src

This expands to a =defun register-weather-tool= function.  The macro:

- Derives the tool name string from the symbol (=system-info= → ="system_info"=)
- Converts each param spec into a =make-properties= call and an =hget= binding
- Wraps the body in =(block handler ...)= so tools can use =return-from handler=
- Generates =(declare (ignore input))= when there are no parameters

**** Parameter specs

Each parameter: =(name type description &key required-p enum default)=

- =name=: Symbol used as the binding variable.  Converted to underscored string for the JSON schema (=action-name= → ="action_name"=).
- =type=: JSON Schema type string (="string"=, ="integer"=, ="boolean"=, ="object"=).
- =:required-p t=: Marks the parameter as required.
- =:enum ("a" "b")=: Restricts to enumerated string values.
- =:default VALUE=: Default value for =hget= when the LLM omits the parameter.

**** Overriding the tool name

When the registration function name doesn't match the tool name string, use
=:tool-name= to override:

#+begin_src lisp
(define-tool log-inspector
    (:description "Inspect daemon logs." :tool-name "daemon_logs")
  ...)
;; Defines register-log-inspector-tool, registers as "daemon_logs"
#+end_src

**** =register-tool (name description schema handler)=

Low-level tool registration.  Prefer =define-tool= for new tools.

**** =register-all-tools ()=

**** =channel-safe-tool-defs ()=

Return the restricted tool list for external channel sessions.  Reads
=[agent] channel_allowed_tools= from config; falls back to the default safe
set (=weather=, =time=, =ephemeris=, =system_info=, =rss=, =battery=,

=:session-type :channel=.

**** =sanitize-user-input (text &key max-length source)=

Sanitise user input before it enters the agent loop:
- Strips control characters (preserves LF, TAB, CR).
- Truncates to =max-length= (warns if truncated).
- Detects injection markers; writes =agent.input.injection_markers= audit event if found.
- Returns the sanitised string or =NIL=.

=max-length= defaults to =+max-channel-input-length+= (8 000).  Use
=+max-direct-input-length+= (32 000) for trusted local clients.  =source= is
a label string for log/audit messages.

Called automatically by =run-agent= / =run-agent/stream= for all sessions.

*** =git= Tool (~skills/builtins/git.lisp~)

The =git= tool provides local git repository and file access, including
the ability to read and write files and make commits.  This is the
primary mechanism for Crichton to modify its own source code.

Gated by two independent config flags:
- =[git] enable = true= — enables read-only operations
- =[git] allow_write = true= — additionally enables write operations

*Configuration:*
#+begin_src toml
[git]
enable = true
allowed_repos = ["/home/user/devel/crichton"]
allow_write = true
#+end_src

- =allowed_repos= is an explicit allowlist (empty = deny all, no default)
- =allow_write= defaults to false; must be set explicitly to permit mutations
- No credentials required — accesses the local filesystem directly

| Parameter      | Type    | Required             | Description                                              |
|----------------+---------+----------------------+----------------------------------------------------------|
| =action=       | string  | yes                  | Action to perform (see below)                            |
| =repo_path=    | string  | for most actions     | Absolute path to git repository (must be in =allowed_repos=) |
| =file_path=    | string  | for file actions     | File path relative to repo or absolute                   |
| =content=      | string  | for =write_file=     | Content to write                                         |
| =paths=        | string  | for =stage=/=unstage= | Comma-separated paths                                    |
| =message=      | string  | for =commit=         | Commit message                                           |
| =branch_name=  | string  | for =create_branch=/=checkout= | Branch name                                  |
| =ref=          | string  | no                   | Git ref (=HEAD=, =main=, =HEAD~3=) for =diff=/=show=    |
| =count=        | integer | no                   | Number of log entries (default 20)                       |
| =all_branches= | boolean | no                   | Include remote tracking branches (=branches= action)     |
| =staged=       | boolean | no                   | Show staged-only changes (=diff= action)                 |

*Read-only actions* (require =enable = true=, =allowed_repos=):
- =status= — =git status --short --branch=
- =log= — recent commits (=--oneline=), filterable by path/branch
- =diff= — working tree or staged changes, optionally against a ref
- =branches= — list branches (optionally including remotes)
- =worktrees= — =git worktree list=
- =read_file= — read a file's current on-disk contents
- =show= — read a file at a specific commit (=git show ref:path=)
- =config_status= — report enabled state and configuration

*Write actions* (also require =allow_write = true=):
- =write_file= — write content to a file (creates or overwrites; parent dirs created)
- =stage= — =git add -- paths=
- =unstage= — =git restore --staged -- paths=
- =commit= — =git commit -m message=
- =create_branch= — =git checkout -b name [from-ref]=
- =checkout= — =git checkout branch=

*Security properties:*
- =allowed_repos= allowlist with canonical path comparison (resolves symlinks, rejects =..= traversal)
- Write gate is independent of read gate — set them separately
- All git commands run via =sb-ext:run-program= with explicit argv (no shell)
- Branch names validated against shell metacharacters before use
- =git= tool excluded from the default channel tool allowlist

*** Rust Example

3. Write skill code in =src/lib.rs=:

#+begin_src rust
use serde::{Deserialize, Serialize};

#[derive(Deserialize)]
struct Input {
    message: String,
}

#[derive(Serialize)]
struct Output {
    result: String,
}

// Memory management for JSON ABI
#[no_mangle]
pub extern "C" fn alloc(size: usize) -> *mut u8 {
    let mut buf = Vec::with_capacity(size);
    let ptr = buf.as_mut_ptr();
    std::mem::forget(buf);
    ptr
}

#[no_mangle]
pub extern "C" fn dealloc(ptr: *mut u8) {
    unsafe { drop(Vec::from_raw_parts(ptr, 0, 0)); }
}

// Main entry point
#[no_mangle]
pub extern "C" fn process(input_ptr: *const u8, input_len: usize) -> i64 {
    // Parse input JSON
    let input_bytes = unsafe {
        std::slice::from_raw_parts(input_ptr, input_len)
    };

    let input: Input = match serde_json::from_slice(input_bytes) {
        Ok(i) => i,
        Err(_) => return 0, // Error: return null pointer
    };

    // Process
    let output = Output {
        result: format!("Processed: {}", input.message),
    };

    // Serialize output
    let output_bytes = serde_json::to_vec(&output).unwrap();
    let output_len = output_bytes.len();
    let output_ptr = alloc(output_len);

    unsafe {
        std::ptr::copy_nonoverlapping(
            output_bytes.as_ptr(),
            output_ptr,
            output_len
        );
    }

    // Return (ptr, len) as packed i64
    ((output_ptr as u64) << 32) | (output_len as u64)
}
#+end_src

*** Coding Practices

**** Prefer =defclass= over =defstruct=

**Always use =defclass= instead of =defstruct=** for data types that may evolve.

**Rationale:**
- SBCL rejects incompatible struct redefinitions during hot-reload
- CLOS classes update existing instances automatically via =update-instance-for-redefined-class=
- Enables iterative development without daemon restart
- No performance penalty in practice

**When to use =defstruct=:**
Only for truly fixed-layout, performance-critical types that will never change. In practice: almost never.

**Example:**

#+begin_src lisp
;; BAD: Will break hot-reload if you add a slot later
(defstruct my-data
  (field1 nil)
  (field2 0))

;; GOOD: Safe for hot-reload
(defclass my-data ()
  ((field1 :initarg :field1
           :initform nil
           :accessor my-data-field1)
   (field2 :initarg :field2
           :initform 0
           :accessor my-data-field2)))

;; Constructor changes from make-my-data to:
(make-instance 'my-data :field1 "value" :field2 42)
#+end_src

**History:**
- cricht-rcl: Original conversion of core structs to classes
- cricht-6d7: Skill registry inadvertently reintroduced defstruct
- cricht-p6i: Policy bead documenting this preference

See AGENTS.org for more development notes.

*** Configuration Sections

**** =[llm]=

- =provider= (string): LLM provider ("anthropic")
- =model= (string): Default model to use
- =max_tokens= (integer): Maximum tokens in response

**** =[logging]=

- =level= (string): Log level (debug, info, warn, error)

**** =[sessions]=

- =encryption= (boolean): Encrypt session storage
- =retention_days= (integer): Auto-delete sessions after N days

**** =[scheduler]=

- =enabled= (boolean): Enable background task scheduler

**** =[battery]=

- =thresholds= (list of integers or single integer): Battery percentage thresholds for proactive alerts.  Default: =[20, 10, 5]=.
- =interval= (integer): Battery polling interval in seconds.  Default: =300=.

AC adapter status is detected via =ac-adapter-online-p= (reads
=ACAD/online= from sysfs).  The =battery-status-plist= includes an
=:ac-online= key and =battery-report= prints the AC power state.

Monitoring state (enabled/disabled, interval) persists across restarts via
encrypted storage.  Monitoring is re-enabled automatically on startup if it
was active when the daemon last shut down.

**** =[system]=

Configures continuous system monitoring.  When enabled (via the =system_info=
agent tool), the daemon polls system metrics at the configured interval and
sends push notifications when thresholds are first crossed.

- =mem_alert_percent= (integer): Alert when memory used >= this percent.  Default: =90=.
- =cpu_alert_load= (float): Alert when the 1-minute load average divided by CPU count >= this value.  Default: =2.0= (200% per core).
- =disk_alert_percent= (integer): Alert when any monitored disk is >= this percent full.  Default: =90=.
- =temp_alert_celsius= (float): Alert when any thermal zone reaches this temperature (°C).  Default: =85.0=.
- =interval= (integer): Polling interval in seconds.  Default: =300= (5 minutes).

Each alert fires once per condition onset — it will not repeat until the
metric recovers and crosses the threshold again.  Alert state is reset when
monitoring is stopped.  Monitoring state persists across daemon restarts via
encrypted storage (=~/.crichton/data/=).

**** =[discord]=

- =enabled= (boolean): Enable Discord channel adapter
- =bot_token_credential= (string): Name of credential containing bot token
- =allowed_user_ids= (list of strings): Discord user IDs allowed to use the bot.  Empty list means all users are allowed (open access).

**** =[agent]=

- =channel_allowed_tools= (list of strings): Tool names available to external channel sessions (Discord, etc.).  Defaults to the read-only safe set: =weather=, =time=, =ephemeris=, =system_info=, =rss=, =battery=, =resource_usage=.  Override to customise.

** Encryption Keys

- Location: =~/.crichton/age-identity.txt=
- Permissions: 0600
- Format: age secret key (bech32)

**IMPORTANT**: Back up =age-identity.txt=. If lost, encrypted credentials and sessions are unrecoverable.

* Storage Systems

** Overview

Crichton has multiple storage layers:

| System              | Location                        | Encryption | Purpose                        |
|---------------------+---------------------------------+------------+--------------------------------|
| Credential Store    | =~/.crichton/credentials/*.age= | age        | API keys, tokens               |
| Session Store       | =~/.crichton/sessions/*.age=    | age        | Conversation history           |
| Persistent Storage  | =~/.crichton/data/*.age=        | age        | Daemon state (meters, RSS)     |
| Skill KV Store      | =~/.crichton/skills-kv/*.age=   | age        | Per-skill persistent data      |
| Scheduler Tasks     | =~/.crichton/tasks/*.age=       | age        | Scheduled task definitions     |
| Logs                | =~/.crichton/logs/daemon.jsonl= | none       | Structured JSON logs (secrets redacted) |

** Persistent Storage (storage/store.lisp)

Namespace-based KV store for daemon state.

*** Namespaces

- =meters=: Token usage statistics
- =rss=: RSS seen-item tracking
- =scheduler=: Task persistence
- =knowledge=: Future local knowledge base

*** Format

Each namespace is a separate =.age= file containing JSON:

#+begin_src json
{
  "version": 1,
  "namespace": "meters",
  "updated_at": "2026-02-13T15:30:00Z",
  "data": {
    "anthropic:input-units": 12345,
    "anthropic:output-units": 6789
  }
}
#+end_src

*** Lifecycle

1. **Startup**: =preload-storage= loads all =*.age= files
2. **Runtime**: Operations modify in-memory cache
3. **Dirty tracking**: Modified namespaces marked dirty
4. **Shutdown**: =flush-all-storage= saves dirty namespaces
5. **Auto-persist**: Some systems (meters, RSS) auto-flush after writes

** Skill KV Store (skills/kv-store.lisp)

Per-skill isolated KV store with quotas.

*** Features

- Per-skill namespace isolation
- Write-through cache
- Quota enforcement (max keys, max total bytes)
- Age encryption

*** Location

=~/.crichton/skills-kv/<skill-name>.age=

*** Access

Only accessible from WASM via =kv_get= and =kv_set= host functions.

** Session Store (sessions/store.lisp)

Conversation session persistence.

*** Features

- Optional age encryption (configurable)
- Session retention policy (auto-delete old sessions)
- Per-session files

*** Location

=~/.crichton/sessions/<session-id>.age= (or =.json= if unencrypted)

*** Format

JSON array of messages:
#+begin_src json
[
  {"role": "user", "content": "Hello"},
  {"role": "assistant", "content": "Hi there!"}
]
#+end_src

** Scheduled Task Persistence (~skills/builtins/task-store.lisp~)

Age-encrypted persistence for user-defined scheduled tasks.  Allows tasks
scheduled via the agent (=schedule_every=, =schedule_daily=, =schedule_once=)
to survive daemon restarts.

*** What Persists

- Tasks whose names begin with =user:= (user-defined tasks).
- The scheduler tool auto-prefixes names with =user:= when scheduling via
  the agent.

System tasks (battery monitor, RSS monitors) are re-registered from config
at startup and do not use this store.

*** Location

=~/.crichton/tasks/user-tasks.age= — age-encrypted JSON.

*** File Format

A versioned JSON envelope wrapping an array of task records:

#+begin_src json
{
  "version": 1,
  "saved_at": "2026-03-21 10:00:00",
  "tasks": [
    {
      "name": "user:rss:lobsters",
      "kind": "every",
      "action_name": "skill:rss-monitor:rss:lobsters",
      "interval_seconds": 1800,
      "daily_hour": null,
      "daily_minute": null,
      "next_ut": 4070908800
    }
  ]
}
#+end_src

Note: =next_ut= is a Common Lisp universal-time integer (seconds since
1900-01-01), not a Unix timestamp.

*** Daemon Lifecycle Integration

| Phase        | Call                                             |
|--------------+--------------------------------------------------|
| Startup      | =restore-user-tasks= after scheduler starts     |
| Shutdown     | =persist-user-tasks= in =persist-all-storage=   |
| Agent tool   | Automatic: every =schedule-every/daily/at= call via the scheduler tool triggers a =persist-user-tasks= |

The startup integration is in =daemon/lifecycle.lisp=.

*** Restore Behaviour

During =restore-user-tasks=:

1. Loads task plists from the encrypted file.
2. For each task, looks up the action by =action-name= in =*schedulable-actions*=.
3. *Missing action* → logs a WARN-level message, task is skipped.
4. *Past one-shot task* → logged at INFO, silently skipped.
5. *Recurring tasks* (=:every=, =:daily=) → rescheduled from their stored =next-ut=
   (which may be in the past; the scheduler will fire them on the next tick).

*** API Reference

**** =user-task-p (task)= → boolean

Returns T if =task= has a name beginning with ="user:"=.

**** =persist-user-tasks ()= → count

Saves all live user tasks to =user-tasks.age=.  Returns the number of tasks
saved.  Thread-safe (holds =*scheduler-lock*=).

#+begin_src lisp
(crichton/skills:persist-user-tasks)   ; => 3
#+end_src

**** =restore-user-tasks ()=

Loads saved tasks and re-registers them with the scheduler.  Call after
=start-scheduler= on daemon startup.  Handles missing actions and past
one-shots gracefully.

**** =load-user-tasks ()= → list of plists

Low-level: reads and decrypts the task file, returns a list of raw plists.
Returns NIL if the file does not exist.

**** =save-user-tasks (tasks)= → count

Low-level: encrypts and writes =tasks= (filtered to user tasks) to disk.

**** =list-unrestorable-tasks ()= → list of plists

Reads the persisted task file and returns tasks whose =action-name= is not
currently registered in =*schedulable-actions*=.  These tasks would be
silently skipped on the next daemon restart.

#+begin_src lisp
(crichton/skills:list-unrestorable-tasks)
;; => ((NAME "user:old-rss" KIND :EVERY ACTION-NAME "skill:old-feed" ...))
#+end_src

**** =export-user-tasks ()= → JSON string

Returns a human-readable (unencrypted) JSON string of all currently-scheduled
user tasks.  Reads live from the in-memory scheduler, not from disk.  Useful
for debugging and inspection.

#+begin_src lisp
(format t "~A" (crichton/skills:export-user-tasks))
;; Prints prettified JSON with exported_at, task_count, and task details
#+end_src

*** Agent Tool Actions

The =scheduler= tool exposes two maintenance actions:

| Action               | Description                                        |
|----------------------+----------------------------------------------------|
| =persist=            | Manually save live user tasks to disk immediately  |
| =list_unrestorable=  | List tasks that would be lost on restart           |

Example usage from =crichton-client=:

#+begin_src bash
crichton-client --eval '(crichton/skills:list-unrestorable-tasks)'
crichton-client --eval '(crichton/skills:persist-user-tasks)'
crichton-client --eval '(format t "~A~%" (crichton/skills:export-user-tasks))'
#+end_src

* Security

** Threat Model

*** Assumptions

1. **User's local system is trusted**: File permissions, systemd, age encryption
2. **Daemon process is privileged**: Has access to credentials, LLM APIs
3. **Runner process is semi-trusted**: Executes WASM but enforces capabilities
4. **WASM skills are untrusted**: Potentially malicious or buggy

*** Threats Mitigated

- **Malicious WASM skills**: Sandboxed, no direct system access
- **Network exfiltration**: HTTP allowlist enforced by capabilities
- **Credential theft**: Encrypted at rest, mediated access
- **Session snooping**: Optional encryption, file permissions
- **Privilege escalation**: WASM has no syscall access
- **Prompt injection via external channels**: Input sanitisation layer strips control chars, enforces length limits, detects and audit-logs injection markers; Discord sessions use a restricted tool allowlist excluding privileged operations
- **Secret leakage via channel responses**: LLM responses are redacted of known secret patterns before being sent to external channels (Discord etc.)

*** Threats NOT Mitigated

- **Local root access**: root can read age-identity.txt and decrypt everything
- **Memory forensics**: Keys briefly in RAM during encryption/decryption
- **Side-channel attacks**: Not hardened against timing attacks
- **Denial of service**: No rate limiting on local socket
- **Sophisticated LLM jailbreaks**: Injection marker detection is heuristic, not a complete defence against determined adversarial prompts

** Sandboxing

*** WASM Isolation

- **Runtime**: wasmtime (Mozilla-maintained, production-grade)
- **No syscalls**: WASM has no access to OS syscalls
- **Capability manifests**: Declare required permissions upfront
- **Host function mediation**: All external access via controlled host functions

*** Capability Enforcement

Capabilities declared in =skill.toml= and enforced by runner:

- =http_domains=: Allowlist of HTTP request targets
- =max_memory_mb=: Memory limit
- =max_cpu_seconds=: CPU time limit (future)
- =kv_store.max_keys=: KV storage quota
- =kv_store.max_total_bytes=: KV size quota

Example enforcement (=runner/skill-context.lisp=):
#+begin_src lisp
(defun check-http-domain (url manifest)
  (let ((allowed (getf manifest :http-domains)))
    (unless (some (lambda (domain)
                    (search domain url))
                  allowed)
      (error "HTTP request to ~A not allowed" url))))
#+end_src

** Encryption

*** Age Encryption

All sensitive data encrypted with =age= (https://age-encryption.org/):

- **Identity**: =~/.crichton/age-identity.txt= (age secret key)
- **Algorithm**: X25519 (ECDH) + ChaCha20-Poly1305
- **Key derivation**: scrypt (for passphrase-protected keys)

*** What's Encrypted

- Credentials (API keys, tokens)
- Sessions (optional, configurable)
- Persistent storage (meters, RSS state)
- Skill KV stores
- Scheduler tasks

*** What's Not Encrypted

- Logs (secrets redacted before logging)
- Config file (no sensitive data)

*** Key Backup

*CRITICAL*: Back up =~/.crichton/age-identity.txt=.  Without it, all encrypted data is lost.

#+begin_src bash
# Backup identity
cp ~/.crichton/age-identity.txt /secure/backup/location/

# Restore identity
cp /secure/backup/location/age-identity.txt ~/.crichton/
chmod 600 ~/.crichton/age-identity.txt
#+end_src

** Secrets Handling

*** Redaction

Logs automatically redact secrets before writing:

- Bearer tokens (regex: =Bearer [A-Za-z0-9._-]+=)
- API keys (regex: =sk-[A-Za-z0-9]{20,}=)
- Passwords, tokens (context-aware)

Implementation: =logging/logger.lisp=, =redact-string=

*** Credential Mediation

Skills cannot directly access credentials. Must request via manifest:

#+begin_src toml
[capabilities]
requires_credential = "openweathermap-api-key"
#+end_src

Daemon mediates access:
1. Runner requests credential via skill context
2. Daemon validates skill's manifest
3. Daemon decrypts credential
4. Daemon injects into skill context (not WASM memory)
5. Skill calls =get_credential()= host function

*** Secure Deletion

Sensitive buffers are zeroed before deallocation:

#+begin_src lisp
(crichton/crypto:with-secure-buffer (buffer size)
  ;; Use buffer
  ...)
;; Buffer zeroed on exit
#+end_src

** Prompt Injection Mitigations

External inputs (Discord messages, RSS feed content, tool results) flow into
LLM prompts.  A malicious input could attempt to override the system prompt
or cause the agent to invoke dangerous tools.  Defence-in-depth is applied at
several layers.

*** Input Sanitisation (~agent/sanitize.lisp~)

Applied to all user input before it enters the agent loop:

- *Control character stripping*: null bytes and non-printable characters removed
  (LF, TAB, CR preserved).
- *Length limits*: channel sessions capped at 8 000 chars; direct local clients
  capped at 32 000 chars.  Operator is warned when truncation occurs.
- *Injection marker detection*: heuristic regex scan for patterns such as
  "ignore previous instructions", =[INST]=, =<|system|>=, etc.  Matches write
  an =agent.input.injection_markers= audit event and a WARN-level log entry.
  Input is still processed — the operator is informed, not the attacker.

#+begin_src lisp
;; Applied automatically in run-agent / run-agent/stream
(sanitize-user-input text :max-length +max-channel-input-length+ :source "discord")
#+end_src

*** Per-Channel Tool Allowlist

When =run-agent= is called with =:session-type :channel= (set automatically for
all RPC requests that carry =channel_type=channel=), the tool list is restricted
to a safe subset via =channel-safe-tool-defs=:

Default allowed set: =weather=, =time=, =ephemeris=, =system_info=, =rss=,
=battery=, =resource_usage=.

Excluded by default: =amp_code=, =amp_test=, =amp_check=, =orgmode=,
=git=, =raindrop=, =github=, =pushover=, =hoobs=, =scheduler= write operations.

Configurable via =[agent] channel_allowed_tools= in =config.toml=.

*** Amp Prompt Sanitisation (~skills/builtins/amp-orchestrator.lisp~)

Text embedded in Amp CLI prompts (task descriptions, context, test output) is
sanitised via =sanitize-for-amp-prompt= before the Amp subprocess is invoked:

- Control characters stripped.
- Length caps: 10 000 chars for description, 5 000 for context, 8 000 for
  embedded test output.
- Injection markers detected and audit-logged as =amp.prompt.injection_markers=.

This specifically prevents a malicious test failure output (e.g. from an RSS
item routed to =amp_test=) from injecting instructions into Amp.

*** Outbound Channel Response Filtering

Before a response is sent to an external channel, =redact-channel-output=
applies the same redaction patterns used for log sanitisation.  If any
redaction fires, a WARN entry is written so the operator knows the LLM
generated a potentially sensitive response.

** Audit Logging

All agent interactions logged to =~/.crichton/logs/daemon.jsonl=:

- User messages
- Agent responses
- Tool invocations
- Skill executions
- Errors and exceptions

Structured audit events (level =AUDIT=) emitted for security-relevant
operations:

| Event type                        | When emitted                                             |
|-----------------------------------+----------------------------------------------------------|
| =agent.input.injection_markers=   | Injection patterns detected in user input                |
| =agent.tool.call=                 | Every tool invoked by the agent loop                     |
| =amp.invoke=                      | Amp CLI invoked (includes policy, exit code, changed files) |
| =amp.invoke.denied=               | Amp invocation blocked by policy                         |
| =amp.prompt.injection_markers=    | Injection patterns in Amp prompt inputs                  |

Log format (JSON Lines):
#+begin_src json
{
  "timestamp": "2026-02-13T15:30:00Z",
  "level": "INFO",
  "logger": "crichton/agent",
  "message": "Invoking tool: weather"
}
#+end_src

Audit event example:
#+begin_src json

** Add a New Built-in Skill

1. Create =skills/builtins/my-skill.lisp=
2. Define skill functions:
   #+begin_src lisp
   (defun my-skill-plist ()
     ;; Return plist of results

     ;; Return formatted string
     (format nil "Result: ~A" (my-skill-plist)))
   #+end_src

3. Register as agent tool in =agent/tools.lisp=:
   #+begin_src lisp
   (register-tool "my-skill"
                  "Description of what my-skill does"
                  #'my-skill-tool-handler

   #+end_src

4. Add to =crichton.asd=:
   #+begin_src lisp
   (:module "skills"
    :components (...
                 (:module "builtins"
                  :components (...
                               (:file "my-skill")))))
   #+end_src

   #+begin_src lisp
   (asdf:load-system :crichton :force t)
   (crichton/agent:register-all-tools)
   #+end_src

** Add a New LLM Provider

1. Create =llm/my-provider.lisp=
2. Define provider class:
   #+begin_src lisp
   (defclass my-provider (llm-provider)
     ((endpoint :initform "https://api.example.com")))
   #+end_src

3. Implement protocol methods:
   #+begin_src lisp
   (defmethod send-message ((provider my-provider) messages
                            &key system max-tokens temperature
                                 tools tool-choice)
     ;; HTTP request to provider API
     ;; Return plist: (:content blocks :stop-reason :end-turn
     ;;               :usage (:input-tokens N :output-tokens N)
     ;;               :id "..." :model "...")
     ...)

   ;; Optional: streaming support (for real-time TUI display)
   (defmethod stream-message ((provider my-provider) messages on-event
                              &key system max-tokens temperature
                                   tools tool-choice)
     ;; HTTP request with streaming (e.g., SSE)
     ;; Call (funcall on-event '(:type :delta :text "..."))
     ;;   for each text chunk
     ;; Call (funcall on-event '(:type :done :stop-reason ...))
     ;;   on completion
     ;; Return same plist as send-message
     ...)
   #+end_src

   If =stream-message= is not implemented, the default method signals
   =llm-feature-not-supported= and =run-agent/stream= falls back to
   =send-message= automatically.

4. Register provider:
   #+begin_src lisp
   (crichton/llm:register-provider "my-provider" 'my-provider)
   #+end_src

5. Update config:
   #+begin_src toml
   [llm]
   provider = "my-provider"
   model = "gpt-4"
   #+end_src

*** Token Usage Metering API (~skills/builtins/token-usage.lisp~)

The metering module tracks resource consumption for any named service.
LLM calls are metered automatically via the =:around= methods in
=llm/protocol.lisp=.

**** Core functions

- =record-usage (meter-name tier input-units output-units)= — Record a usage
  event.  =tier= is typically the model name (e.g. ="claude-sonnet-4-20250514"=).
  Updates the in-memory meter immediately; persistence is fire-and-forget.

- =ensure-meter (name)= — Get or create a named meter.

- =meter-snapshot (name)= — Return a plist with totals, burn rate, and cost.

- =aggregate-snapshot ()= — Combined snapshot across all meters.

- =meter-recent (name &optional n)= — Last N usage records as plists.

**** Persistence

Meters are persisted under the ="meters"= storage namespace, keyed as
="all-meters"=, whose value is a hash-table mapping meter name to a plist:

#+begin_src lisp
(crichton/storage:store-get "meters" "all-meters")
;; => #<hash-table "llm" => (:name "llm" :total-input 128000 :total-output 16000
;;                           :total-cost 0.624d0 :call-count 42 ...) >
#+end_src

- =persist-meter (name)= — Write one meter to storage immediately.
- =save-meters ()= — Snapshot all meters to storage.
- =load-meters ()= — Restore meters from storage (called on daemon startup).
- =*auto-persist-meters*= — When true (default), =record-usage= triggers
  =persist-meter= automatically (in a background thread).

**** Pricing tables

Model pricing is registered at load time:

#+begin_src lisp
(register-pricing "llm"
  '(("claude-sonnet-4-20250514"  :input 3.00d0 :output 15.00d0)
    ("claude-opus-4-20250514"    :input 15.00d0 :output 75.00d0)
    ...))
#+end_src

Cost estimates are computed per call and accumulated in =meter-total-cost=.

** Manually Flush Storage

#+begin_src lisp
;; Flush all dirty namespaces
(crichton/storage:flush-all-storage)

;; Flush specific namespace
(crichton/storage:flush-namespace "meters")
#+end_src

* Writing WASM Skills

This section is a focused developer guide for authoring WASM skills from
scratch.  The [[*Skill Development][Skill Development]] section above covers the manifest format
and Rust/JSON-ABI details; this section emphasises the rationale, the
directory layout, the WAT quick-start path, and the full host function
reference.

** Why WASM?

| Property      | Benefit                                                                   |
|---------------+---------------------------------------------------------------------------|
| Sandboxed     | No direct filesystem, network, or environment variable access             |
| Portable      | Write once in any WASM-capable language (Rust, C, AssemblyScript, WAT)   |
| Signed        | Ed25519 signatures verify code integrity before execution                 |
| Capability-based | Explicit =skill.toml= manifest declares what the skill is allowed to do |
| Isolated      | Runs in a separate runner process — cannot crash or compromise the daemon |

This matches Design Principle 1: /Skills must be isolated/.

** Skill Bundle Structure

#+begin_src plantuml :file skills.png :exports results
@startsalt
{T
  + ~/.crichton/skills/my-skill/
  ++ skill.toml      (Manifest — required)
  ++ skill.wasm      (Compiled WASM binary — required)
  ++ skill.wat       (WAT source — optional, human-readable)
}
@endsalt
#+end_src

The =skill.toml= manifest must be present.  =skill.wasm= is the runnable
binary.  =skill.wat= is optional source; commit it for version control but
do not ship it to end users.

** Quick Start: WAT (5 minutes)

WAT (WebAssembly Text format) is the human-readable form of WASM and
requires no compiler toolchain beyond =wat2wasm=.

*Install tools:*
#+begin_src bash
sudo apt install wabt   # Debian/Ubuntu
brew install wabt       # macOS
#+end_src

*Create the skill directory:*
#+begin_src bash
mkdir -p ~/.crichton/skills/meaning-of-life
cd ~/.crichton/skills/meaning-of-life
#+end_src

*Write =skill.toml=:*
#+begin_src toml
[skill]
name = "meaning-of-life"
version = "1.0.0"
author = "you@example.com"
description = "Returns 42"
signature = ""

[capabilities]
http_domains = []
max_memory_mb = 16
max_cpu_seconds = 5
#+end_src

*Write =skill.wat=:*
#+begin_src wat
(module
  ;; Import Crichton's log host function
  (import "env" "log" (func $log (param i32 i32 i32)))

  ;; Export linear memory (required for host functions that read strings)
  (memory (export "memory") 1)

  ;; Embed a string at offset 0 in the data section
  (data (i32.const 0) "Hello from WASM!")

  ;; Export the entry-point function
  (func (export "main") (result i32)
    ;; log(INFO=1, offset=0, len=16)
    (call $log
      (i32.const 1)   ;; level: INFO
      (i32.const 0)   ;; string starts at offset 0
      (i32.const 16)) ;; "Hello from WASM!" is 16 bytes
    ;; Return the answer
    (i32.const 42)))
#+end_src

*Compile:*
#+begin_src bash
wat2wasm skill.wat -o skill.wasm
#+end_src

*Test from the SLIME/SLY REPL:*
#+begin_src lisp
(crichton/wasm:run-wasm-with-host-fns
  "(module
     (import \"env\" \"log\" (func $log (param i32 i32 i32)))
     (memory (export \"memory\") 1)
     (data (i32.const 0) \"Hello from WASM!\")
     (func (export \"main\") (result i32)
       (call $log (i32.const 1) (i32.const 0) (i32.const 16))
       (i32.const 42)))"
  "main")
;; => 42  (also logs "Hello from WASM!" at INFO level)
#+end_src

See =examples/skills/meaning-of-life/= for the complete working example.

** Host Functions Reference

Skills have no direct system access.  All external capabilities come through
host functions imported from the ="env"= module.

*** =env.log(level: i32, ptr: i32, len: i32)=

Log a UTF-8 string from WASM linear memory.

| Argument | Meaning                                      |
|----------+----------------------------------------------|
| =level=  | 0 = debug, 1 = info, 2 = warn, 3 = error     |
| =ptr=    | Byte offset in WASM memory where string starts |
| =len=    | Length of string in bytes                    |

Returns nothing (void).  *Status*: Implemented.

WAT import:
#+begin_src wat
(import "env" "log" (func $log (param i32 i32 i32)))
#+end_src

*** =env.kv_get(key_ptr: i32, key_len: i32, out_ptr: i32, out_cap: i32) -> i32=

Retrieve a value from the skill's private key-value store.

| Argument   | Meaning                           |
|------------+-----------------------------------|
| =key_ptr=  | Pointer to key string in memory   |
| =key_len=  | Length of key string in bytes     |
| =out_ptr=  | Pointer to output buffer          |
| =out_cap=  | Capacity of output buffer in bytes |

Returns: bytes written to =out_ptr=, or -1 if key not found.
*Status*: Implemented (via =crichton/skills:kv-get=, stored in =~/.crichton/skills-kv/=).

*** =env.kv_set(key_ptr: i32, key_len: i32, val_ptr: i32, val_len: i32) -> i32=

Store a key-value pair in the skill's private storage.

Returns: 0 on success, negative error code on failure.
*Status*: Implemented.

*** =env.kv_delete(key_ptr: i32, key_len: i32) -> i32=

Delete a key from the skill's private storage.

Returns: 0 on success, -1 if not found.
*Status*: Implemented.

*** =env.kv_exists(key_ptr: i32, key_len: i32) -> i32=

Check whether a key exists in the skill's private storage.

Returns: 1 if exists, 0 if not.
*Status*: Implemented.

*** =env.kv_list(prefix_ptr: i32, prefix_len: i32, out_ptr: i32, out_cap: i32) -> i32=

List all keys in the skill's store that match =prefix= (empty prefix = all
keys).  The output is a newline-delimited list of key strings.

Returns: bytes written to =out_ptr=, or -1 on error.
*Status*: Implemented.

*** =env.http_request(method_ptr: i32, method_len: i32, url_ptr: i32, url_len: i32, headers_ptr: i32, headers_len: i32, body_ptr: i32, body_len: i32) -> i32=

Make an HTTP request.  The URL must match a domain declared in
=capabilities.http_domains= in =skill.toml=.

Returns: HTTP status code, or negative error code.
*Status*: Host function registered; full implementation pending (returns 403
until complete).  URL allowlist enforcement logic is in
=runner/skill-context.lisp:check-http-allowlist=.

*** =env.get_secret(name_ptr: i32, name_len: i32) -> i32=

Retrieve a secret from Crichton's encrypted credential store.  The skill
must declare =requires_credential = "credential-name"= in its =skill.toml=.

Returns: pointer to secret value in WASM memory, or -1 if not found or not
allowed.
*Status*: Host function registered; full mediation implementation pending.

** Capability Enforcement

The runner process checks all capability constraints before and during skill
execution:

- =http_domains=: Each HTTP request target is validated against the allowlist.
- =max_memory_mb=: Passed to wasmtime as a memory limit at instantiation time.
- =kv_store.max_keys= and =kv_store.max_total_bytes=: Enforced by the KV
  store layer before every write.
- Credentials: The skill must declare =requires_credential= in its manifest;
  the daemon validates this before injecting the credential.

Skills that violate their declared capabilities receive error return codes
from host functions — they are not killed.  This is intentional: a skill
returning an error code is safer than an abrupt WASM trap.

* State Files Subsystem

The =crichton/state= module manages the three identity/continuity files
that are injected into the LLM system prompt on every turn, plus a daily
journal that is searchable on-demand.

** Why Plain Text?

- Org-mode is the project's documentation standard.
- The LLM reads prose, not data structures — no schema drift.
- Human-editable in Emacs without tooling.
- Can be age-encrypted at rest (the infrastructure already exists).

** File Layout

#+begin_src plantuml :file state.png :exports results
@startsalt
{T
  + ~/.crichton/state/
  ++ SOUL.org          (Persona, values, behavioural tone)
  ++ USER.org          (User identity and preferences)
  ++ MEMORY.org        (Curated long-term facts — main sessions only)
  ++ journal/
  +++ 2026-03-02.org   (Today's raw daily log — append-only)
  +++ 2026-03-01.org   (Yesterday's daily log)
}
@endsalt
#+end_src

Directory permissions: =0700=.  File permissions: =0600=.

** File Definitions

*** SOUL.org — Persona and Values

Who is Crichton?  How should it behave?  Loaded on every turn.  The
user writes this file; the agent may suggest edits but must ask first.
A minimal template is created on first boot if the file does not exist.

*** MEMORY.org — Curated Long-Term Memory

What does Crichton need to remember across sessions?  Auto-injected in
main sessions.  *Not* injected in channel (Discord) sessions or subagent
contexts — personal facts must not leak to those contexts.

The agent writes to this file when it learns durable facts (preferences,
ongoing projects, lessons learned).  It is curated prose, not a raw log.

*** USER.org — User Context

Who is the human?  Loaded alongside SOUL.org on every turn.

*** journal/ — Daily Logs

Raw, append-only daily notes.  Never auto-injected (would blow the context
window).  Searchable on-demand via the =memory_search= agent tool.  One
file per day: =journal/YYYY-MM-DD.org=.

* TUI Architecture

This section describes the TUI client's internal architecture for
developers working on =crichton-tui=.  For user-facing documentation see
the [[*Architecture Overview][Architecture Overview]] section.

** TEA (The Elm Architecture) via cl-tuition

The TUI is built on =cl-tuition=, a Common Lisp port of Go's Bubble Tea
framework.  It implements the TEA pattern:

#+begin_src plantuml :file tea-pattern.png :exports results
@startuml
left to right direction
rectangle "Event" as E
rectangle "update-message\n(CLOS method)" as U
rectangle "Model\n(CLOS)" as M
rectangle "Command\n(zero-arg lambda;\nruns on thread pool)" as C
rectangle "view\n(fn)" as V
rectangle "Terminal I/O" as T

E --> U
U --> M : new state
U --> C : side effect
M --> V
V --> T
@enduml
#+end_src

- *Model*: A =tui-model= CLOS instance holding all application state (chat
  history, input buffer, scroll position, status, notifications, session ID,
  terminal dimensions, spinner).
- *Messages*: CLOS instances created via =tui:defmessage=.  The main loop
  dispatches each message to =tui:update-message= specialised on the model
  and message type.
- *View*: A pure function from model to terminal output (ANSI escape strings).
  Called after every =update-message= invocation.
- *Commands*: Zero-argument lambdas returned (optionally) from
  =update-message=.  The framework runs them on a thread pool so the main
  loop never blocks on I/O.

** Message Types

#+begin_src lisp
;; Incoming from the daemon I/O thread
(tui:defmessage daemon-response-msg   ; normal chat response
  ((id error-p text session)))
(tui:defmessage daemon-notification-msg ; push notification
  ((kind text source)))
(tui:defmessage daemon-disconnected-msg ())

;; User actions
(tui:defmessage send-chat-msg ((text)))

;; Framework messages (from cl-tuition)
;; tui:key-msg, tui:window-size-msg, tui:tick-msg
#+end_src

** Update Cycle

1. User presses Enter → =tui:key-msg= dispatched to =update-message=
2. Model updated (new =chat-entry= pushed, input cleared, status → =:waiting=)
3. =update-message= returns a command: a lambda that writes NDJSON to the
   daemon socket on a pool thread
4. Daemon responds asynchronously; the I/O thread reads the response and
   calls =(tui:send program (make-instance 'daemon-response-msg ...)=)
5. =daemon-response-msg= dispatched to =update-message=: appends assistant
   entry to model, clears spinner, status → =:idle=
6. =view= re-renders the complete frame

For streaming responses, =chat_delta= push messages arrive before
=chat_done= and are accumulated into a provisional =chat-entry= that is
updated on each delta.

** Threading Model

#+begin_src plantuml :file threading.png :exports results
@startuml
rectangle "**Main Thread** (cl-tuition loop)\n\n  · Terminal I/O and rendering\n  · Input decoding\n  · update-message dispatch\n  · view → terminal output" as Main
rectangle "**Daemon I/O Thread** (bt:make-thread)\n\n  · Persistent connection to\n    ~/.crichton/daemon.sock\n  · Reads NDJSON lines in a loop\n  · Injects daemon-response-msg\n    and daemon-notification-msg" as IO

Main -down-> IO : tui:send (thread-safe)
@enduml
#+end_src

*Invariant*: The daemon I/O thread never touches terminal state directly.
All state mutations happen on the main thread via =tui:send=.

** How the TUI Connects to the Daemon

The TUI reuses the same NDJSON-over-Unix-socket protocol as =crichton-client=.
Two extensions are used:

- ={op: "chat", stream: true}=: The daemon pushes =chat_delta= and
  =chat_done= messages before (and instead of) the normal response.
- ={op: "subscribe"}=: The daemon registers the connection for unsolicited
  =notify= push messages (battery alerts, RSS items, scheduler events).

The daemon I/O thread loop dispatches both types of unsolicited messages
by checking the ="op"= field of every incoming NDJSON line.  See
=tui/client.lisp:start-daemon-reader= for the dispatch logic.

* Coding Conventions

** =defclass= over =defstruct= (Hot-Reload Safety)

*Always use =defclass= instead of =defstruct=* for data types that may
evolve.

*Rationale*: SBCL rejects incompatible struct redefinitions at hot-reload
time, forcing a daemon restart.  CLOS classes update existing instances
automatically via =update-instance-for-redefined-class=, enabling live
iteration.  Use =defstruct= only for truly fixed-layout,
performance-critical types that will never change.  In practice: almost
never.

See the history in beads cricht-rcl (original conversion), cricht-6d7
(inadvertent regression), cricht-p6i (policy bead).

** Hot-Loading vs Restart

*Never restart the daemon* to pick up code changes.  Restart loses all
in-memory context (session history, channel state, scheduler queue).

Use SBCL hot-loading instead:

#+begin_src lisp

(load "/path/to/file.lisp")            ; Single file
(crichton/agent:register-all-tools)    ; After tool handler changes
#+end_src

Or via the command line with =swank-client= (see =AGENTS.org= for the
one-liner).
