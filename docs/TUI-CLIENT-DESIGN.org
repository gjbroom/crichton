* Motivation

The current =crichton-client= uses cl-readline over a synchronous
request/response NDJSON protocol.  This has three fundamental limitations:

1. *No colour* — readline strips ANSI escape sequences; Crichton's responses
   arrive as flat text.
2. *No terminal control* — no cursor movement, no layout, no panels.  The
   output is a dumb scroll.
3. *No push notifications* — the client can only display output in response to
   user input.  The daemon has no way to notify the user of scheduled-task
   results, battery alerts, RSS items, or Discord messages unless the user
   happens to send a chat message.

A TUI client resolves all three while preserving the microkernel architecture:
the TUI binary is still a thin process that speaks NDJSON over the daemon
socket.

* Design Goals

- Full ANSI colour (256-colour and true-colour foreground/background)
- Adaptive layout that reflows on terminal resize (=SIGWINCH=)
- Asynchronous notification display (daemon pushes events to client)
- Persistent input area separate from scrolling output
- Markdown-ish rendering of LLM responses (bold, italic, code blocks, lists)
- Sixel graphics display in terminals that support it
- Zero daemon-side changes required for basic functionality
- Minimal dependencies (no ncurses; pure ANSI escape sequences)

* Library Choice: cl-tuition

** Why cl-tuition

| Criterion             | cl-tuition                     | croatoan / cl-charms          |
|-----------------------+--------------------------------+-------------------------------|
| Architecture          | TEA (Elm) with CLOS            | imperative ncurses wrapper    |
| ncurses dependency    | None — pure ANSI               | Requires libncurses           |
| Colour support        | 256 + truecolour, adaptive     | ncurses colour pairs          |
| Layout primitives     | =join-vertical=, =join-horizontal=, =place=, borders | Manual positioning |
| Mouse support         | Yes (press, release, drag, scroll) | Via ncurses                |
| Async commands        | Built-in (=tui:tick-cmd=, custom) | Manual thread management   |
| Widgets               | Spinner, progress, list, table, text-input | Minimal built-in  |
| Markdown rendering    | Built-in (Glamour-inspired)    | None                          |
| Licensing             | MIT                            | MIT                           |
| Maturity              | Active (2025–2026), 129 stars  | Stale                         |
| SBCL compatible       | Yes (bordeaux-threads)         | Yes                           |

cl-tuition is the clear winner.  It is modelled on Go's Bubble Tea (the
library that powers the Go TUIs Gord admired), but in Common Lisp with
CLOS.  No ncurses dependency means no CFFI complexity and trivially
reproducible builds via =save-lisp-and-die=.

** Dependencies added to =crichton-tui.asd=

- =cl-tuition= — TUI framework
- =shasht= — JSON (already used by =crichton-client=)
- =bordeaux-threads= — threading (transitive via cl-tuition)

No =cl-readline= dependency.

The TUI client is a separate ASDF system (=crichton-tui=) and a separate
=save-lisp-and-die= binary, just like =crichton-client= today.

** Threading model

Two threads:

1. *Main thread* — runs the cl-tuition program loop (terminal I/O,
   rendering, input decoding).
2. *Daemon I/O thread* — maintains a persistent connection to
   =daemon.sock=, reads NDJSON lines, and injects messages into the
   tuition program loop via =tui:send=.

The daemon I/O thread never touches terminal state directly.

** =tui/protocol.lisp= — Daemon wire protocol

Reuses the same NDJSON-over-Unix-socket protocol as the current client.
Provides:

- =connect-daemon= — opens Unix socket, returns stream
- =write-message= / =read-message= — NDJSON encode/decode
- =send-request= — fire-and-forget (response arrives asynchronously on
  the reader thread)

This is essentially the same code as =client/client.lisp= minus readline,
refactored for async I/O.

*** Adaptive layout

The view function computes regions as follows:

- *Title bar*: 1 line, full width
- *Input area*: 1–3 lines (grows with wrapped input text), anchored to bottom
- *Status bar*: 1 line, above input
- *Chat area*: remaining height, scrollable

On resize (=tui:window-size-msg=), width and height update in the model,
and the next =view= call reflows everything.  Long assistant responses are
word-wrapped to =model-width - 4= (2 chars padding each side).

*** LLM response rendering

Assistant text is rendered with lightweight Markdown-to-ANSI conversion:

- =**bold**= → =tui:bold=
- =*italic*= → =tui:italic=
- =`inline code`= → bright-yellow on dark background
- Fenced code blocks (=```=) → bordered panel with syntax-appropriate colour
- Bullet lists → indented with styled bullets
- Numbered lists → indented with styled numbers

** =tui/sixel.lisp= — Sixel graphics support

*** Approach

Sixel is a DEC terminal protocol for inline raster graphics.  It is
supported by: xterm (=+sixel=), foot, mlterm, WezTerm, Contour, and
others (see https://www.arewesixelyet.com/).

The TUI client does not generate sixel data itself.  Instead:

1. The daemon's LLM response or skill output may include an image
   reference (a file path or base64-encoded image data) in a
   structured content block.
2. The TUI client converts the image to sixel format by shelling out to
   =img2sixel= (from libsixel) if available.
3. The sixel data is cached and emitted at the appropriate position
   during rendering.

*** Detection

#+begin_src lisp
(defun sixel-supported-p ()
  "Query the terminal for sixel support via DA1 (Device Attributes).
   Returns T if the terminal reports sixel capability."
  ;; Send DA1: ESC [ c
  ;; Parse response for ";4;" (sixel graphics attribute)
  ;; Fallback: check TERM env var for known sixel-capable terminals
  ...)

(defun img2sixel-available-p ()
  "Return T if img2sixel is on PATH."
  (zerop (nth-value 2 (uiop:run-program
                        '("which" "img2sixel")
                        :ignore-error-status t))))
#+end_src

*** Integration

Sixel data is emitted as raw escape sequences between rendered text
lines.  cl-tuition's pure-string rendering means we can embed sixel
escape sequences directly in the view output — the terminal will
interpret them.

The =view= function checks =entry-sixels= on each =chat-entry= and,
if the terminal supports sixel, injects the cached sixel data at the
appropriate scroll position.  On non-sixel terminals, a fallback
=[Image: filename]= placeholder is shown.

*** Daemon I/O thread

A background thread reads from the daemon socket and injects messages:

#+begin_src lisp
(defun start-daemon-reader (stream program)
  "Spawn a thread that reads NDJSON from STREAM and sends
   messages into the tuition PROGRAM."
  (bt:make-thread
   (lambda ()
     (handler-case
         (loop for msg = (read-message stream)
               while msg
               do (let ((result (gethash "result" msg)))
                    (tui:send program
                              (make-instance 'daemon-response-msg
                                             :id (gethash "id" msg)
                                             :text (gethash "text" result)
                                             :session (gethash "session_id" result)
                                             :error-p (not (gethash "ok" msg))))))
       (error ()
         (tui:send program (make-instance 'daemon-disconnected-msg)))))
   :name "daemon-reader"))
#+end_src

*** Sending chat

When the user presses Enter, =update-message= on =tui:key-msg= checks for
the Enter key, creates a =send-chat-msg=, and returns a command that
performs the socket write on a separate thread (so the TUI loop never
blocks on I/O):

#+begin_src lisp
(defmethod tui:update-message ((model tui-model) (msg tui:key-msg))
  (let ((key (tui:key-msg-key msg)))
    (cond
      ((eq key :enter)
       (let ((text (model-input-text model)))
         (unless (zerop (length (string-trim '(#\Space) text)))
           (push (make-instance 'chat-entry
                                :role :user
                                :text text
                                :time (get-universal-time))
                 (model-messages model))
           (setf (model-input-text model) ""
                 (model-status model) :waiting
                 (model-scroll-offset model) 0)
           (values model (send-chat-cmd text)))))
      ;; ... other key handling: scrollback, Ctrl-C, etc.
      )))
#+end_src

*** Entry point

#+begin_src lisp
(defun main ()
  (let ((args (rest sb-ext:*posix-argv*)))
    (handler-case
        (if args
            ;; One-shot mode: send args as chat, print response, exit
            (one-shot-chat (format nil "~{~A~^ ~}" args))
            ;; Interactive TUI mode
            (tui-repl))
      (error (c)
        (format *error-output* "~A~%" c)
        (sb-ext:exit :code 1)))))
#+end_src

* RPC Protocol Extensions (Daemon Side)

Basic TUI operation requires *zero daemon changes* — the existing
request/response protocol works as-is, with the reader thread waiting
for responses asynchronously instead of synchronously.

However, to unlock push notifications, the RPC protocol needs a small
extension.  This is a *Phase 2* enhancement.

** Phase 2: Server-initiated notifications

Add a =subscribe= RPC operation:

#+begin_src javascript
// Client → Daemon
{"id": 42, "op": "subscribe", "events": ["notification", "status"]}

// Daemon → Client (unprompted, no correlation ID)
{"op": "notify", "kind": "battery", "text": "Battery at 15%", "source": "battery-monitor"}
{"op": "notify", "kind": "rss", "text": "New item in HN feed: ...", "source": "rss-monitor"}
{"op": "notify", "kind": "scheduler", "text": "Task 'backup' completed", "source": "scheduler"}
#+end_src

The daemon reader thread already loops on =read-message=.  It would
dispatch notification messages as =daemon-notification-msg= instances
instead of =daemon-response-msg=.

This requires:
- A new =subscribe= op handler in =daemon/rpc-server.lisp=
- Per-connection notification delivery in =handle-rpc-connection=
- A notification registry that skills/monitors can post to

** Phase 3: Streaming responses

The LLM protocol already defines =stream-message= (currently
unimplemented for Anthropic).  Once implemented daemon-side, the RPC
protocol can stream response chunks:

#+begin_src javascript
// Daemon → Client (multiple lines for one request)
{"id": 42, "op": "chat_delta", "text": "Currently"}
{"id": 42, "op": "chat_delta", "text": " in Victoria"}
{"id": 42, "op": "chat_done", "result": {"text": "...", "session_id": "..."}}
#+end_src

The TUI would render partial text as it arrives, giving the user immediate
feedback.  This is a natural extension of the notification mechanism.

** Minimum terminal requirements

- 80×24 minimum (graceful degradation at smaller sizes)
- ANSI escape sequence support (virtually all modern terminals)
- UTF-8 encoding

* Relationship to Existing Client

The readline client (=crichton-client=) remains available for simple
scripting and one-shot queries.  The TUI client is a separate binary
for interactive use.  Both coexist; the user chooses which to run.

Eventually the TUI client may become the default interactive interface,
but the readline client will not be removed — it is useful for
=crichton-client "what time is it?"= style shell scripting.

* Open Questions

1. *Scrollback buffer limit* — how many chat entries to keep in memory
   before paging to disk?  Suggest 1000 entries, with older entries
   evicted to an on-disk file.

2. *Multi-line input* — should Shift+Enter insert a newline in the
   input area?  cl-tuition's text-input widget may need extension for
   this.  Alternative: detect terminal Shift+Enter support via
   =CSI 27;2;13~= (kitty keyboard protocol).

3. *Image format support* — should we support only sixel, or also
   kitty's graphics protocol for broader compatibility?  Suggest
   sixel-first with kitty protocol as a Phase 4 stretch goal.

4. *Session persistence* — the TUI could save/restore the visual chat
   history locally (separate from daemon session storage) for instant
   re-display on reconnect.
