* Parallel Development with Git Worktrees

For working on multiple features simultaneously without conflicts, use *git worktrees*. Each worktree is a separate working directory pointing to a different branch, all sharing the same =.git= repository.

** Creating Worktrees for Parallel Features

When spawning background agents to work on different features in parallel:

#+begin_src bash
# From main repo: /home/gjbroom/devel/crichton
cd /home/gjbroom/devel/crichton

# Create worktree for feature 1
git worktree add ../crichton-readline feature/readline-support

# Create worktree for feature 2
git worktree add ../crichton-monitoring feature/system-monitoring

# Create worktree for feature 3
git worktree add ../crichton-orgmode feature/orgmode-skill

# List all worktrees
git worktree list
#+end_src

Each worktree:
- Has its own working directory (=../crichton-<feature>=)
- Points to its own feature branch
- Shares the same =.git= repository (no duplication)
- Agents can work independently without clobbering each other

** Workflow for Parallel Development

1. *Create feature branches and worktrees*:
   #+begin_src bash
   git checkout master
   git branch feature/new-feature
   git worktree add ../crichton-new-feature feature/new-feature
   #+end_src

2. *Spawn background agent in worktree*:
   - Agent works in =../crichton-new-feature= directory
   - Agent commits to =feature/new-feature= branch
   - No conflicts with other agents in other worktrees

3. *When agent completes*:
   #+begin_src bash
   # Switch to main repo
   cd /home/gjbroom/devel/crichton

   # Merge completed feature
   git checkout master
   git merge feature/new-feature

   # Clean up worktree
   git worktree remove ../crichton-new-feature
   git branch -d feature/new-feature
   #+end_src

4. *Push merged changes*:
   #+begin_src bash
   git push origin master
   #+end_src

** Benefits

- *True parallelism*: Multiple agents work simultaneously without conflicts
- *Isolation*: Each feature development is independent
- *Efficiency*: Shared =.git= means no repository duplication
- *Clean merges*: Feature branches merge cleanly into master
- *Productivity*: Work on 3-5 features in parallel

** Important Notes

- Always create worktrees from the main repository directory
- Worktree paths are relative to current directory (use =../crichton-<name>=)
- Clean up worktrees after merging: =git worktree remove <path>=
- Don't delete worktree directories manually; use =git worktree remove=
- Each worktree needs its own branch (can't have two worktrees on same branch)

* Spawning Subagents for Parallel Work

When using the Task tool to spawn subagents for parallel development, follow these critical guidelines to ensure successful execution.

** Permission Requirements

*CRITICAL*: Background agents (=run_in_background=true=) cannot request permissions interactively. If they hit permission-required operations without approval, they will fail silently and may produce fabricated "success" summaries instead of actual work.

*Solutions*:
1. *Preferred*: Use permissive permission mode for parallel development sessions
2. *Alternative*: Spawn agents without =run_in_background= (they can still run in parallel via multiple tool calls in one message)
3. *Fallback*: Pre-approve specific tools before spawning background agents

** Required Prompt Template

*ALL* subagent prompts MUST include this error handling directive:

#+begin_example
**CRITICAL ERROR HANDLING:**
If you encounter permission errors, tool failures, or cannot complete any part of your task:
1. STOP immediately
2. Report the specific failure in your response
3. Do NOT fabricate success summaries
4. Do NOT claim work is complete when it isn't
5. Do NOT produce "aspirational" documentation of work you didn't do

If you hit a blocker, say so explicitly. Honesty about failures is mandatory.
#+end_example

** Working in Worktrees

#+begin_example

** Verification After Completion

*Never trust agent completion summaries blindly*. Always verify:

#+begin_src bash
# Check commits exist
cd /path/to/worktree && git log --oneline -5

# Check files were actually created/modified
ls -lh path/to/expected/file.lisp

# Check file contents
grep -n "expected-function" path/to/file.lisp
#+end_src

If an agent claims success but verification fails, the agent *did not actually complete the work* - it fabricated a success summary. This happens when agents hit permission errors in background mode.

* Landing the Plane (Session Completion)

*When ending a work session*, you MUST complete ALL steps below. Work is NOT complete until =git push= succeeds.

*MANDATORY WORKFLOW:*

1. *File issues for remaining work* - Create issues for anything that needs follow-up
2. *Run quality gates* (if code changed) - Tests, linters, builds
3. *Update issue status* - Close finished work, update in-progress items
4. *PUSH TO REMOTE* - This is MANDATORY:
   #+begin_src bash
   git pull --rebase
   bd sync
   git push
   git status  # MUST show "up to date with origin"
   #+end_src
5. *Clean up* - Clear stashes, prune remote branches
6. *Verify* - All changes committed AND pushed
7. *Hand off* - Provide context for next session

*CRITICAL RULES:*
- Work is NOT complete until =git push= succeeds
- NEVER stop before pushing - that leaves work stranded locally
- NEVER say "ready to push when you are" - YOU must push
- If push fails, resolve and retry until it succeeds
